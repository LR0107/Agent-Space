# EvolveR: Self-Evolving LLM Agents through an Experience-Driven Lifecycle

**ICLR 2026**

**EvolveR让智能体把过去做题的成功和失败总结成“解题原则”，在下一次做题时主动查阅这些原则，并通过强化学习逐渐学会更有效地使用经验。**

## 一、研究背景

现在的大语言模型智能体已经可以：

- 进行多步推理；
- 调用搜索引擎、数据库等工具；
- 完成复杂问答任务。

但它们通常把每次任务看成一个完全独立的事件。上一道题做对了什么、做错了什么，往往不会真正影响下一道题。

传统的检索增强生成（RAG）主要解决：

> “模型不知道某个事实怎么办？”

它让模型访问外部文档、网页或知识库。

但EvolveR关注的是另一个问题：

> “模型已经做过许多任务，能否从自己的行为结果中总结出更好的做事方法？”

### 现有经验学习方法的不足

论文将已有方法概括为三类。

**第一类：无状态执行**

每完成一个任务，就直接丢弃交互过程。

完全不积累经验。

**第二类：直接保存原始轨迹**

把过去完整的“思考—搜索—回答”过程保存起来，遇到新问题时检索相似案例。

问题是：

- 原始轨迹很长；
- 容易包含无关细节；
- 对新问题的泛化能力有限；
- 模型容易照抄旧答案，而不是理解旧答案为什么有效。

**第三类：由外部强模型总结经验**

使用更强的教师模型总结智能体的轨迹。

问题是：

- 需要额外的模型和成本；
- 教师总结出的策略可能与当前智能体自身的思维方式不一致；
- 经验通常只是作为提示加入上下文，没有真正改变智能体的策略。

EvolveR提出第四种范式：**智能体自己产生轨迹、自己总结原则、自己使用原则，再通过强化学习更新自身策略。**

## 二、研究方法

论文主要研究三个问题：

1. **LLM智能体能否从自己的成功和失败轨迹中提炼出可复用的策略？**
2. **这些策略能否在后续任务中有效指导推理和工具调用？**
3. **能否通过强化学习，让智能体逐渐学会何时检索经验、如何使用经验？**

**1. 离线经验自蒸馏**

什么是“经验自蒸馏”？

智能体完成任务后，会留下完整的交互轨迹，例如：

```
问题→ 思考→ 检索经验→ 搜索外部知识→ 阅读结果→ 继续推理→ 给出答案
```

在离线阶段，模型参数被暂时冻结。智能体使用**自己的模型**分析这些轨迹，把具体行为总结成抽象、简短的原则。

成功轨迹会产生**指导性原则**，例如：

> “比较两个对象时，应先分别获取双方的信息，再进行判断。”

失败轨迹会产生**警示性原则**，例如：

> “不要根据名称相似性直接判断两个人是同一人物，应核验身份信息。”

**2. 经验库的去重、合并和筛选**

智能体可能从不同轨迹中总结出相似原则。

如果全部保留，经验库会迅速膨胀。

因此，EvolveR设计了一套经验管理机制。

**第一步：同一问题内部去重**

对同一道问题生成的多条原则进行语义比较，只保留具有代表性的版本。

**第二步：与已有经验匹配**

对于一条新原则：

1. 先使用向量相似度，从经验库中找出最接近的已有原则；
2. 再让LLM判断二者是否在语义上等价。

处理方式是：

- 不等价：作为新原则加入；
- 等价：不新建原则，而是把新轨迹合并到原原则下面。

**第三步：动态评分**

每条原则都会记录：

- 被使用了多少次；
- 使用后成功了多少次。

> 一条原则被调用得越多，而且调用后越容易得到正确答案，它的分数就越高。

系统会周期性删除低于阈值的原则，避免经验库被低质量、空泛或误导性的内容占据。

Experience Base：一个基于Milvus实现、位于Qwen模型参数之外、专门保存和检索经验的长期记忆组件。

**3. 在线交互**

在线阶段采用类似 **Think–Act–Observe** 的多轮交互方式。

智能体主要可以执行三种动作：

| 动作                    | 作用               |
| --------------------- | ---------------- |
| `<search experience>` | 在内部经验库中检索策略原则    |
| `<search knowledge>`  | 在外部知识库或搜索引擎中查找事实 |
| `<answer>`            | 输出最终答案并结束任务      |

这里最关键的区别是：

- 外部知识检索告诉智能体“事实是什么”；
- 经验检索告诉智能体“应该怎么查、怎么推理”。

例如，智能体面对一个人物比较问题时，可以先执行：

```
<search experience>如何处理人物比较问题？</search experience>
```

经验库可能返回：

> “比较前分别验证两个对象的身份和属性，不要只搜索其中一个。”

随后，智能体再据此设计外部搜索步骤。

因此，经验不是直接替智能体回答问题，而是改变它的**搜索顺序、推理结构和决策方式**。

**Top-3语义相似度**

**4.策略进化：使用强化学习更新模型**

只把经验放进数据库还不够。

模型还需要学习：

- 什么情况下应该搜索经验；
- 应该使用什么检索词；
- 检索到多条经验后应采用哪一条；
- 什么时候继续搜索外部知识；
- 什么时候应该结束并回答。

作者使用 GRPO（Group Relative Policy Optimization）进行强化学习。

GRPO更新是本地部署的 Qwen2.5

GPT-4o的冷启动数据是为了产生“格式正确的轨迹”，而不是教授模型任务知识。

如果直接对原始Qwen做GRPO，这样得到的轨迹结构混乱，奖励信号也很差。冷启动相当于先教会它“怎么参加训练”，再通过GRPO学习“怎样做得更好”。

**奖励函数**

总奖励由两部分组成：

**结果奖励**

根据最终答案是否与标准答案完全匹配进行二元评分：

- 答对：得到奖励；
- 答错：没有奖励。

**格式奖励**

鼓励智能体形成完整、合理的行为轨迹，包括：

- 正确使用 `<think>`；
- 使用经验检索或知识检索；
- 最终输出 `<answer>`；

也就是说，模型不仅需要“最后碰巧答对”，还要形成比较规范的推理和工具使用过程。

**GRPO怎样训练？**

对于同一个问题，系统一次采样多条不同轨迹。

GRPO对同一个问题运行多次。

例如生成2种做法：

- 有的先查经验再查知识；
- 有的完全不查经验；

系统比较这一组轨迹的奖励，提升高奖励行为的概率，降低低奖励行为的概率。

因此，它学到的并不只是“如何回答问答题”，还包括：如何利用自己的经验来完成问答题。

主体实验基于Qwen2.5系列：0.5B、1.5B、3B、7B；GPT-4o只在冷启动阶段生成少量示范数据。

**真正持续产生轨迹、总结经验和接受GRPO更新的都是Qwen2.5**

### 工作流程

**做任务—总结经验—使用经验—根据结果继续改进**

- **在线交互**  
  智能体通过思考、检索外部知识和回答问题来完成任务，同时记录完整的操作轨迹。
- **离线自蒸馏**  
  冻结模型参数，让智能体分析成功和失败的轨迹，将其总结为可复用的“策略原则”
- **维护经验库**  
  对新原则进行语义去重、合并和动态评分；有效经验被保留，低质量经验被删除。
- **经验检索与应用**  
  遇到新任务时，智能体通过 `<search experience>` 检索相关原则，再根据这些原则决定如何推理和搜索外部知识。
- **强化学习更新**  
  根据答案是否正确以及交互流程是否规范，使用GRPO更新模型，使其逐渐学会何时检索经验、如何使用经验。

## 三、实验设计

### 实验设置

**模型**

实验使用 **Qwen2.5** 系列模型，覆盖：

- 0.5B
- 1.5B
- 3B
- 7B

主实验重点报告3B和7B模型，0.5B和1.5B主要用于分析方法随模型规模变化的表现。

**数据集**

作者使用了7个问答数据集，并明确区分了域内和域外任务。

| 类型  | 数据集                   | 用途          |
| --- | --------------------- | ----------- |
| 域内  | Natural Questions（NQ） | 训练、构建经验库和测试 |
| 域内  | HotpotQA              | 训练、构建经验库和测试 |
| 域外  | TriviaQA              | 只用于泛化测试     |
| 域外  | PopQA                 | 只用于泛化测试     |
| 域外  | 2WikiMultiHopQA       | 只用于多跳泛化测试   |
| 域外  | MuSiQue               | 只用于多跳泛化测试   |
| 域外  | Bamboogle             | 只用于困难、对抗性测试 |

域内任务的训练集用于形成经验；域外任务不参与经验库构建

**Baseline**

实验设置了三类基线。

**提示与检索方法**

- Direct Inference
- Chain-of-Thought
- RAG
- IRCoT
- Search-o1

这些方法一般不更新模型参数，主要依赖提示、推理链或外部检索。

**监督微调方法**

- 标准SFT
- Rejection Sampling

这类方法从静态的专家或筛选数据中学习。

强化学习方法

- DeepSeek-R1相关模型
- Search-R1相关模型

其中Search-R1与EvolveR最接近，因为二者都会通过强化学习训练模型调用搜索工具。

区别在于：

- Search-R1主要学习如何搜索**外部事实**；
- EvolveR额外学习如何总结和利用**内部经验**。

**评价指标**

Exact Match（EM）：只有模型答案经过标准化后与标准答案完全一致，才算正确。

**训练流程**

实验大致分为两个阶段。

**阶段一：冷启动**

作者从NQ和HotpotQA训练集中选择约700条带有思维链和工具交互的轨迹，对模型进行LoRA微调。

目的不是直接教给模型大量答案，而是让模型先掌握规范的输出形式

**阶段二：强化学习进化**

主要设置包括：

- 每个训练步骤采样128个问题；
- 每个问题生成8条轨迹；

- 学习率为 1×10−6；
- mini-batch大小为128；
- 使用8张A100 GPU；
- 采用Verl框架训练。

### 实验结果

主实验

在3B模型上，EvolveR在以下任务上表现较突出：

- NQ：0.434
- HotpotQA：0.373
- 2Wiki：0.381
- MuSiQue：0.137
- Bamboogle：0.328

在7B模型上：

- NQ：0.462
- HotpotQA：0.411
- TriviaQA：0.620
- PopQA：0.473
- MuSiQue：0.168
- Bamboogle：0.392

模型规模

| 模型   | 平均分   |
| ---- | ----- |
| 0.5B | 0.150 |
| 1.5B | 0.270 |
| 3B   | 0.382 |
| 7B   | 0.417 |

**消融实验**

作者用GPT-4o-mini作为外部教师，与模型自己总结经验进行比较。

| 模型规模 | 自蒸馏       | 教师蒸馏      |
| ---- | --------- | --------- |
| 0.5B | 0.150     | **0.220** |
| 1.5B | 0.270     | **0.290** |
| 3B   | **0.382** | 0.370     |

结论是：

- 对小模型而言，外部强教师明显更可靠；
- 到3B规模时，自蒸馏反而更好。

作者保持训练过程完全相同，只在测试时禁止模型访问经验库。

3B模型的平均分从：0.382→0.340

## 四、总结

### 创新点

不只是“把经验给模型看”，还通过GRPO训练模型学会使用经验

### 不足

自我进化能力明显受基础模型能力限制：小模型不仅做题成功率低，提炼经验、判断经验是否重复、理解经验的能力也较弱。

可能产生错误经验的自我强化：系统会从失败轨迹中提炼警示原则，但如果模型错误理解了失败原因，也可能得到错误结论。

计算成本高：

- 8张A100 80GB；
- 39.4小时；
- 经验库最终约14,000条原则。

### Q&A

**1.为什么还需要GRPO？直接把经验放进Prompt不行吗？**

把经验放进上下文只能让模型“看到经验”，但不能保证模型知道什么时候检索、检索什么、是否采用以及何时继续搜索外部知识。EvolveR把经验检索视为一种可学习的动作，通过GRPO比较同一问题的多条轨迹，使模型逐渐学会有效的工具调用和推理策略。

**2.这篇文章最核心的创新是什么？**

最核心的创新不是提出新基础模型，而是提出了一个**经验驱动的自我演化生命周期**：模型先产生任务轨迹，再从轨迹中提炼策略原则，将原则存入动态经验库，之后在新任务中检索这些原则，并通过GRPO学习如何有效使用经验。它将“经验生成、经验管理、经验使用和参数更新”统一在一个系统中。
