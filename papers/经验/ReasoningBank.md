# ReasoningBank: Scaling Agent Self-Evolving with Reasoning Memory

**ICLR 2026**

**大语言模型智能体在不断执行任务时，怎样把过去的成功经验和失败教训转化为可复用的策略，从而越用越聪明？**

## 一.研究背景

目前的大语言模型智能体已经可以完成很多需要多步操作的任务，例如浏览网页、查询订单。

但这类智能体通常把每个任务当作一个全新的问题。

例如，智能体第一次查询用户最早的订单时，可能因为只查看了“最近订单”而失败。下一次遇到类似问题时，它仍然可能犯同样的错误，因为上一次的失败没有形成可复用的经验。

因此，现有智能体主要存在三个问题：

1. 容易重复过去的错误；
2. 无法复用相似任务中的有效方法；
3. 随着任务数量增加，能力不会自然提升。

已有的智能体记忆方法大致有两类。

| 方法    | 保存的内容           | 主要问题                |
| ----- | --------------- | ------------------- |
| 轨迹记忆  | 保存完整的思考、点击和操作过程 | 内容长、噪声多，而且容易与特定网页绑定 |
| 工作流记忆 | 从成功任务中抽取固定操作流程  | 通用性有限，通常无法利用失败经验    |

例如，保存一段完整的网页操作轨迹，相当于记录：

> 点击订单 → 点击第2页 → 找到某个订单。

这种记忆对完全相同的页面可能有效，但页面结构或任务一变化，就不一定适用。

ReasoningBank 不强调记住具体点击步骤，而是总结更一般的原则，例如：

> 查询“最早”“全部”或“历史记录”时，不能只查看当前可见内容，应检查分页

论文认为，这种**策略级推理记忆**比原始轨迹和固定工作流更容易迁移到新任务。

## **二.研究方法**

可以把 ReasoningBank 理解为一本经验手册。

每完成一个任务，智能体会回答三个问题：

1. 这次任务成功了吗？
2. 成功的原因或失败的原因是什么？
3. 有哪些经验可以用于以后类似的任务？

这些经验被整理后写入 ReasoningBank。执行下一项任务时，系统先从经验手册中找出最相关的几条策略，再让智能体参考这些策略完成任务。

### 记忆结构

每条记忆由三个部分组成：

**Title：标题**

简要概括这条策略，例如：

> 优先从用户账户区域查找个人数据

**Description：描述**

用一句话说明该策略适用于什么情况，例如：

> 当任务询问用户自己的订单、地址或账户信息时，应首先检查个人账户相关入口。

**Content：具体内容**

例如：

> 先寻找“My Account”“Orders”等用户专属区域；如果任务涉及最早、最晚或全部记录，应检查完整订单历史、分页和排序方式，而不是只依赖最近记录。

这种结构既方便人理解，也可以直接放入大模型的系统提示词中。

JSON 形式保存。每次任务对应一条经验记录。

任务查询的 embedding 会预先计算并单独保存，方便后续做相似度检索。每次运行时，记忆池会持久化，并随着任务流**持续增长**。

### 关键步骤

**第一步：记忆检索**

接收到新任务后，系统会：

1. 将当前任务转换成向量；
2. 将它与历史任务进行相似度比较；
3. 找到最相关的历史经历；
4. 取出对应的策略记忆；
5. 把这些记忆加入智能体的系统提示词。

通过余弦相似度进行检索，默认检索最相似的 1 个历史经历。

在智能体每一步操作前，提示词要求它先判断：

> 当前记忆是否适用于这个任务？应该怎样使用？

这样可以避免智能体机械地套用不相关的经验。

**第二步：记忆提取**

任务结束后，系统需要从完整轨迹中总结新经验。

首先，作者使用 **LLM-as-a-Judge**，根据以下信息判断任务是否成功：

- 用户原始要求；
- 智能体完整操作轨迹；
- 网页最终状态；
- 智能体给用户的最终回答。

评审模型只输出两类结果：

```
Success
Failure
```

这里不使用测试集的真实答案作为学习信号。

之后，根据成功或失败采用不同的总结方式：

- **成功轨迹**：提取已经得到验证的有效策略；
- **失败轨迹**：分析错误原因，提取应避免的陷阱、检查方法和修正原则。

每条轨迹最多提取 **3 条记忆**。记忆提取模型与智能体采用相同的主干模型。

**第三步：记忆整合**

新生成的记忆会被直接加入 ReasoningBank。

论文刻意采用了非常简单的整合方法：

- 不进行复杂合并；
- 不进行额外剪枝；
- 不主动删除旧记忆；
- 将新记忆直接追加到记忆池。

故意保持简单，是为了避免复杂记忆管理算法干扰实验，突出“记忆内容设计”的作用。

但这也意味着，长期运行后可能出现记忆数量持续增长、内容重复等问题。

### 为什么要同时学习成功和失败？

这是论文与许多已有记忆方法最关键的区别之一。

假设智能体在查询“最早购买日期”时失败，原因是它只查看了“最近订单”。

传统方法可能直接丢弃这次失败轨迹。

ReasoningBank 则可能提取出：

> 当任务要求最早记录时，应确认当前列表的排序方向，并检查完整历史、分页或“查看全部”入口。

失败本身没有提供正确答案，但提供了一个非常重要的**反事实信号**：

> 原来的做法为什么不可靠，以后应该增加什么检查。

所以 ReasoningBank 不只是保存“应该怎么做”，也保存“为什么某种做法可能失败”。

**反事实信号**可以理解为：

> 根据一次失败，思考“假如当时采用另一种做法，是否可以避免失败”，并把这个差异总结成经验。

### 工作流程

**1.接收新任务**：智能体依次处理网页操作或代码修复任务。

**2.检索相关记忆**：根据当前任务，从 ReasoningBank 中找出最相关的历史策略和失败教训，并加入提示词。

**3.执行任务**：智能体参考这些记忆，与网页进行多步交互。

**4.判断任务结果**：任务结束后，由另一个大模型充当评审，根据完整操作轨迹判断任务是成功还是失败。

**5.提取新经验：**

- 从成功任务中总结有效方法；
- 从失败任务中总结错误原因、风险和改进策略。

**6.更新记忆库**：将新经验整理成“标题、描述、具体策略”的结构，加入 ReasoningBank，供后续任务使用。

## 三.实验设计

### 实验设置

**1. 任务**

作者在三类基准上进行测试。

WebArena：真实风格的网页操作，包含 **684 个任务**

包括信息查询、页面导航、内容修改等。

Mind2Web：网页操作，Mind2Web 共使用 **1341 个测试实例**

SWE-Bench-Verified：软件问题修复，包含 **500 个经过人工验证的软件问题**

**2.主干模型**

- Gemini-2.5-Flash；
- Gemini-2.5-Pro；
- Claude-3.7-Sonnet。

**3.对比方法**

论文设置了三个主要对照组：

| 方法            | 记忆形式              |
| ------------- | ----------------- |
| No Memory     | 不使用任何历史记忆         |
| Synapse       | 保存并检索历史完整轨迹       |
| AWM           | 从成功轨迹中提取工作流       |
| ReasoningBank | 从成功和失败轨迹中提取通用推理策略 |

**4.评价指标**

WebArena

主要使用两个指标：

- **成功率 SR**：完成任务的比例，越高越好；
- **平均步骤数 AS/Step**：每个任务平均需要多少次智能体—环境交互，越低越好。

步骤数用于衡量效率：一个方法即使成功率高，如果需要大量无意义点击，也不够理想。

Mind2Web

使用四个指标：

- **EA**：是否选中了正确的网页元素；
- **AF1**：对元素执行的动作是否正确；
- **SSR，Step Success Rate**：当前步骤的元素和动作是否都正确；
- **SR，任务成功率**：任务中的所有步骤是否全部正确。

其中任务级 SR 最严格：只要中间有一步错误，整个任务就不算完全成功。

SWE-Bench-Verified

- **Issue Resolve Rate**：补丁是否真正修复问题并通过全部测试；
- **平均步骤数**：修复一个问题需要多少次交互。

### 实验结果

**1.WebArena 整体结果**

| 主干模型              | 无记忆   | ReasoningBank |
| ----------------- | ----- | ------------- |
| Gemini-2.5-Flash  | 40.5% | 48.8%         |
| Gemini-2.5-Pro    | 46.7% | 53.9%         |
| Claude-3.7-Sonnet | 41.7% | 46.3%         |

可以看到：

1. ReasoningBank 在三个不同模型上都优于无记忆方法；
2. 提升并不依赖某一个特定模型系列。

以 Gemini-2.5-Flash 为例，平均步骤数从无记忆的 **9.7 步**降到 ReasoningBank 的 **8.3 步。**这说明记忆不仅提高成功率，也能减少无效探索。

**2.软件工程结果**

在 SWE-Bench-Verified 上：

| 模型               | 方法            | 修复率   | 平均步骤 |
| ---------------- | ------------- | ----- | ---- |
| Gemini-2.5-Flash | 无记忆           | 34.2% | 30.3 |
|                  | ReasoningBank | 38.8% | 27.5 |
| Gemini-2.5-Pro   | 无记忆           | 54.0% | 21.1 |
|                  | ReasoningBank | 57.4% | 19.8 |

### **消融实验和进一步分析**

**1.失败经验是否有用？**

作者比较了两种设置：

- 只从成功轨迹中学习；
- 同时从成功和失败轨迹中学习。

在 WebArena-Shopping 上，ReasoningBank：

- 只使用成功轨迹时：**46.5%**；
- 加入失败轨迹后：**49.7%**。

**2.LLM 评审不完全准确怎么办？**

实际测试中，LLM-as-a-Judge 判断成功或失败的准确率约为 **72.7%**。

作者还模拟了不同准确率的评审器。结果表明，在约 70%—90% 的合理范围内，ReasoningBank 的最终性能变化不大。

这意味着系统对一定程度的错误判断具有鲁棒性。

可能的原因是：

- 单条错误记忆不一定会被后续任务检索到；
- 随着经验增加，其他轨迹可能补充或纠正错误经验

## 四.总结

### 创新点

1.把完整操作轨迹提升为可迁移的推理策略

2.把失败轨迹转化为记忆：这是 ReasoningBank 相比很多成功轨迹记忆方法更突出的地方。

### 不足

**1.高度依赖底层模型的判断和归纳能力**

Agent、Judge 和 Memory Extractor 都依赖大语言模型。

一旦 Judge 判断错误，可能出现：

```
失败轨迹 → 被判断为成功 → 错误策略写入记忆
```

**2.记忆管理过于简单，长期使用容易积累噪声**

当前方法主要采用：

- 基于任务文本 embedding 的相似度检索；
- 将新记忆直接追加进记忆库；
- 不主动去重、合并、删除或更新旧记忆。

### Q&A

**1.这篇文章主要解决什么问题？**

它解决的是大语言模型智能体“做完任务却不会积累经验”的问题。传统智能体通常把每个新任务当作独立任务，容易重复过去的错误。ReasoningBank 将历史成功与失败轨迹提炼成可复用的推理策略，在后续任务中检索并使用

**2.为什么不直接保存完整轨迹？**

完整轨迹虽然信息多，但通常很长，包含大量与具体网页、按钮编号和操作顺序相关的噪声。新任务与旧任务不完全相同时，直接复用轨迹容易误导智能体。

**3.这个方法是否非常依赖底层模型能力？**

是的。它依赖模型完成任务、判断成败、分析失败原因、抽象通用策略以及正确使用记忆。如果底层模型无法理解任务，或者经常错误归因，ReasoningBank 也可能把错误经验写入记忆。

**4.LLM-as-a-Judge 判断错误怎么办？**

评审错误可能造成记忆污染，例如把失败轨迹当作成功经验。论文测得评审准确率约为72.7%，并模拟了不同评审准确率，发现系统在70%到90%的范围内表现相对稳定。这表明它能够容忍一定噪声，但并没有从根本上解决错误评审问题。
