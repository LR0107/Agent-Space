# ExpeL: LLM Agents Are Experiential Learners

**AAAI 2024 清华大学**

不修改大语言模型参数，而是让智能体通过做任务、总结经验和回忆相似案例，逐步提高解决新任务的能力

## 一.研究背景

传统的大语言模型主要完成问答、写作和文本生成任务。近年来，研究者开始把 GPT 等模型包装成智能体，让模型能够：

- 搜索信息；
- 操作网页；

这类任务不是一次性生成答案，而是一个连续决策过程：

> 观察环境 → 思考 → 执行动作 → 获得新信息 → 再思考。

论文主要采用 **ReAct** 作为基础智能体框架。

### 现有学习方法存在两个主要问题

**问题一：微调成本高，而且不一定能访问模型参数**

一种常见做法是使用大量任务数据对大语言模型进行微调。但这存在几个问题：

- 需要大量标注数据；
- 需要较高的计算成本；
- 必须能够访问模型参数；
- 可能损害模型原本的通用能力；
- GPT-4、Claude 等闭源模型只能通过 API 使用，无法直接微调其参数。

因此，微调并不适合所有智能体场景。

**问题二：提示学习缺少长期记忆**

另一种方法是在提示词中加入少量示例，让模型通过上下文学习。但模型的上下文长度有限，无法永久保存所有历史经历。

传统智能体往往出现这样的情况：它完成了很多任务，但完成任务后没有把经验保留下来；面对下一项任务时，基本上又从零开始。

### Reflexion只能解决“单个任务内”的学习问题

Reflexion 是论文重点比较的方法之一。它允许智能体在任务失败后进行自我反思，例如：

> “我失败的原因是搜索关键词过于宽泛，下次应该加入商品属性。”

随后，智能体重新尝试同一个任务。

这种方法可以实现**任务内学习**：

- 第一次失败；
- 总结原因；
- 第二次重新尝试；
- 在同一个任务上逐渐改善。

但这些反思通常不会被系统地迁移到其他任务。也就是说，Reflexion 更关注：“如何在这道题上重试成功？”

而 ExpeL 关注的是：“我能否从许多练习题中总结经验，并在新题目上直接做得更好？”

因此，论文试图实现的是一种**跨任务经验学习**。

## 二.研究方法

大语言模型智能体能否在不更新模型参数的情况下，通过自主收集和总结历史经验，提高在新任务上的表现？

具体来说，作者希望智能体同时获得两类能力：

| 学习方式   | 类似人类的行为    | ExpeL中的实现         |
| ------ | ---------- | ----------------- |
| 具体经验记忆 | 回忆以前做过的相似题 | 检索相似的成功轨迹         |
| 抽象规律总结 | 总结通用解题方法   | 从成功与失败案例中提炼自然语言规律 |

论文认为，仅记住案例不够，仅总结规律也不够，两者结合效果最好。

### 步骤

**第一阶段：收集成功和失败经验**

在训练阶段，ExpeL会面对一组练习任务。

对于每个任务，智能体首先使用 ReAct 进行尝试。如果任务失败，则利用 Reflexion分析失败原因，然后再次尝试。

每次尝试产生的完整过程称为一条**轨迹**，其中包含：

- 环境观察；
- 智能体思考；
- 执行动作；
- 环境反馈；
- 最终成功或失败结果。

所有轨迹都会进入一个“经验池”。

**为什么同时保存成功和失败经验？**

因为失败和成功的对比非常有价值。

假设同一个任务中：

- 失败轨迹搜索了过于笼统的关键词；
- 成功轨迹使用了更具体的关键词；
- 两条轨迹进行对比后，智能体可以总结：

> “搜索结果过多时，应当加入任务中的关键限制条件重新组织搜索词。”

因此，失败经验不是简单丢弃，而是被用于理解“哪里做错了”。

**第二阶段：从经验中提取通用规律**

经验收集结束后，ExpeL使用一个大语言模型分析经验池，并生成一组自然语言形式的规则。

论文使用两种方式来提取。

**方法一：比较同一任务的失败轨迹和成功轨迹**

模型同时查看：

- 失败时采取了什么行动；
- 成功时采取了什么行动；
- 两者的关键差异是什么。

这种方法主要用于发现：

- 常见错误；
- 不必要的行动；
- 导致失败的决策方式；
- 更合适的替代策略。

**方法二：分析多个不同任务的成功轨迹**

模型还会同时观察一批成功案例，寻找这些案例共有的规律。

例如，在多个购物任务中，成功轨迹可能都表现出：

- 先确认所有商品属性；
- 搜索结果不准确时重新组织关键词；
- 点击购买前再次核对尺寸、颜色和价格。

这些共同特点会被抽象成一般规则。

**规则的动态维护机制**

ExpeL不是一次性生成规则，而是维护一个可以不断修改的列表。

大语言模型可以执行四种操作：

| 操作         | 含义               |
| ---------- | ---------------- |
| `ADD`      | 添加一条新的           |
| `EDIT`     | 修改已有             |
| `UPVOTE`   | 新经验支持该规则，提高其重要性  |
| `DOWNVOTE` | 新经验不支持该规则，降低其重要性 |

每条规则都有一个重要性计数。

- 新的初始计数为2；
- 被多次验证时，计数增加；
- 被反例否定时，计数减少；
- 计数降到0时，该条被删除。

这种设计的作用是防止智能体因为一两个偶然案例，总结出错误规则。

例如，某次成功轨迹可能让模型形成：

> “寻找厨房物品时，应当先搜索抽屉。”

但后续多个案例表明该规则并不可靠，模型就会降低其权重，最终将它删除或修改为：

> “寻找物品时，应结合物品的用途判断其常见位置。”

**第三阶段：测试时使用规则和相似经验**

在测试阶段，ExpeL面对没有见过的新任务，并且原则上只允许尝试一次。

此时，它会同时获得两类辅助信息。

**第一类：抽象规则**

所有经过筛选的通用规则会被加入提示词中，例如：

> - 搜索失败时，重新检查问题中的实体和限制条件。
> - 如果已有观察足以支持答案，应及时结束搜索。
> - 当前行动没有推动任务进展时，应重新评估状态并选择替代行动。

这些类似于学生整理的“错题规律”和“解题技巧”。

**第二类：相似任务的成功轨迹**

ExpeL还会从经验池中检索与当前任务最相似的成功案例。（不用LLM）

具体实现为：

- 将任务文本转换为向量；
- 使用 FAISS 建立向量数据库；
- 根据任务相似度检索前 k 条成功轨迹；
- 把这些轨迹作为上下文示例提供给智能体。

**和普通RAG的区别**

ExpeL包含检索机制，看起来类似 RAG，但两者的重点不同。

普通 RAG 通常检索：

- 外部文档；
- 人工编写的知识库；

ExpeL检索的是：

> **智能体自己过去生成的成功经验。**

因此，它是让智能体通过实践逐步建立自己的经验库。

---

### 工作流程

- **收集经验**  
  智能体在训练任务中使用 ReAct 执行任务。若失败，就用 Reflexion 分析错误并重新尝试，同时保存成功和失败的完整轨迹。
- **提取经验规律**  
  系统比较同一任务的成功与失败过程，并分析多个成功案例，总结出通用的自然语言规则。这些规则会通过添加、修改、支持和否定机制持续更新。
- **解决新任务**  
  测试时，系统把总结出的规则加入提示词，并从经验库中检索与当前任务最相似的成功案例，作为示例帮助智能体决策。此阶段通常只尝试一次，不再依赖反复重试。

## 三.实验设计

### 实验设置

| 环境       | 任务数量     |
| -------- | -------- |
| HotpotQA | 100      |
| ALFWorld | 134个可解任务 |
| WebShop  | 100      |

**Baseline**

Act：模型只输出行动，不显式生成推理过程。

ReAct：模型交替进行推理和行动

Reflexion：任务失败后进行反思，并重新尝试同一任务。

为了分析不同模块的作用，作者还设计了：

- **ExpeL insights-only**：只使用总结出的规则；
- **ExpeL retrieve-only**：只检索相似成功经验；
- **完整ExpeL**：同时使用规则和经验检索。

**模型**

智能体执行动作时统一使用 gpt-3.5；

默认使用 gpt-4提取规则

通过 LangChain 调用 OpenAI API。论文中提到的 RTX 2080 Ti 并不负责运行 GPT-4；本地设备主要运行环境、向量检索和控制程序。

选择 GPT-4 做洞见提取，是因为这一步要求模型：

- 同时理解多条长轨迹；
- 准确比较成功和失败；
- 遵循 ADD、EDIT、UPVOTE、DOWNVOTE 格式；
- 避免生成过度泛化或虚假的规则。

消融实验也显示，用 GPT-3.5 提取洞见时，性能低于 GPT-4。

**评估方式**

主要指标是**任务成功率**。

不同任务对“成功”的定义不同：

- HotpotQA和FEVER：最终答案完全匹配；
- ALFWorld：在规定步数内完成目标；
- WebShop：购买的商品满足全部指定属性。

---

### 实验结果

完整ExpeL在三个主要环境中都优于基线

| 环境       | ReAct | 完整ExpeL |
| -------- | ----- | ------- |
| HotpotQA | 约28%  | 约39%    |
| ALFWorld | 约40%  | 约59%    |

**迁移学习实验**

作者进一步研究：在一种任务上获得的经验，能否帮助另一种相关任务？

实验设置为：

- 源任务：HotpotQA；
- 目标任务：FEVER。

两者都需要使用 Wikipedia 搜索工具，但任务目标不同：

- HotpotQA要求回答问题；
- FEVER要求验证陈述真假。

作者先从 HotpotQA经验中提取洞见，再使用少量FEVER示例对这些洞见进行调整。

| 方法                 | FEVER成功率 |
| ------------------ | -------- |
| Act                | 58%      |
| ReAct              | 63%      |
| 直接迁移洞见，不使用目标任务示例   | 65%      |
| ExpeL迁移，使用少量目标任务示例 | **70%**  |

**消融实验**

作者比较了三种情况：

- 只分析人工提供的少量示例；
- 使用 ReAct收集经验，但不进行失败后的反思重试；
- 使用 Reflexion收集包含成功和失败对比的多样经验。

结果显示：

- 只依赖原有少量示例，几乎没有明显提升；
- 自动收集更多经验后，性能提高；
- 同时拥有成功和失败轨迹时效果最好。

**模型自动提取的规则优于人工规则**

| 方法              | 成功率     |
| --------------- | ------- |
| ReAct           | 28%     |
| 人工编写            | 32%     |
| GPT-3.5提取       | 32%     |
| 完整ExpeL，GPT-4提取 | **39%** |

作者尝试把 Reflexion生成的自我反思文本也作为规则，但性能从完整ExpeL的39%下降到约29%。

## 四.总结

### 创新点

**提出不更新参数的经验学习方法：** ExpeL不需要训练，也不需要访问模型权重，因此可以直接用于GPT-4、Claude等闭源模型。

**同时结合“抽象规律”和“具体案例”**：

ExpeL同时使用：

- 自然语言形式的通用规则；
- 与当前任务相似的成功轨迹。

实验表明二者具有互补性。

**实现跨任务学习和迁移学习：** ExpeL能够把多个训练任务中的经验用于未见过的测试任务，还能把一种任务中的知识迁移到相关任务。

### 不足

**实验依赖闭源模型**：论文使用了GPT-3.5和GPT-4。闭源API可能存在：

- 成本问题；
- 隐私问题；
- 无法离线部署；
- 模型版本变化。

作者认为后续应测试开源模型。

**长期学习可能导致上下文过长**：当前实验中的规则数量较少，可以全部放进提示词。但如果智能体长期积累成千上万条经验和规则，就无法全部放入上下文。

### Q&A

**1.为什么只检索成功轨迹，不检索失败轨迹？**

测试时加入的轨迹会被大模型当作模仿示范，因此成功轨迹更适合作为上下文示例。失败轨迹可能引导模型重复错误，所以主要用于训练阶段的对比分析。

**2.为什么既需要洞见，又需要轨迹检索？**

洞见提供抽象规律，适合处理与训练任务不完全相同的新情况；成功轨迹提供具体操作范例，适合结构相似的任务。仅有洞见可能过于抽象，仅有检索又可能缺乏泛化能力。论文实验显示，两者结合通常优于只使用其中一个模块。

**3.为什么ExpeL的测试阶段通常只允许尝试一次？**

作者希望模拟现实中的“练习—考试”过程：训练阶段可以反复练习，但测试或部署时可能没有重试机会。例如网页购买、机器人操作或一些高成本决策中，失败后重试不可行

**4.ExpeL是否能够实现真正的终身学习？**

论文中的洞见规模较小，可以全部放入上下文；但长期运行后，经验库和规则库会越来越大，出现上下文超长、经验过时、规则冲突和检索效率下降等问题。要实现终身学习，还需要经验压缩、遗忘机制和分层记忆。

**5.ExpeL是否会受到错误经验污染？**

会。成功轨迹不一定是最优轨迹，模型也可能从偶然成功中总结出错误规则。论文通过规则投票、编辑和删除机制缓解这一问题，但不能完全消除。
