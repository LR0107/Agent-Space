# ProcMEM: Learning Reusable Procedural Memory from Experience via Non-Parametric PPO for LLM Agents

**ICML 2026**

## 一、研究背景

LLM 智能体在解决多步任务时，通常采用类似下面的方式：

1. 读取当前状态；
2. 进行推理；
3. 选择行动；
4. 获得环境反馈；
5. 再次推理。

这种方式在第一次处理任务时是合理的，但当智能体反复遇到类似情况时，它仍然会从头推理。

这会带来三个问题：

- **计算冗余**：重复消耗大量推理和上下文 token；
- **执行不稳定**：每次重新推理都可能产生不同结果；
- **长任务容易累积错误**：任务越长，前面的小错误越容易影响后续步骤。

**参数化学习的局限**

一种解决思路是通过强化学习或微调，把经验写入模型参数。

但这种方法通常存在：

- 训练成本高；
- 需要更新大模型权重；
- 可能造成灾难性遗忘；
- 可能使模型过度适应特定任务，损害通用能力。

因此，论文希望做到：

> **模型参数完全冻结，仅通过外部记忆实现学习。**

**传统外部记忆仍然不够**

现有智能体通常保存以下内容：

- 原始交互轨迹；
- 对成功或失败的总结；
- 反思或经验教训；
- 工作流或知识图结构。

这些更接近情景记忆（episodic memory），相当于一本“过去发生过什么”的历史记录。

使用时，模型仍然需要：

1. 检索过去案例；
2. 阅读和理解案例；
3. 判断当前情况是否相似；
4. 再次推导具体行动。

ProcMEM希望建立的是**程序性记忆（procedural memory）**，即直接保存“在什么情况下，应该按照什么步骤做”。

| 记忆类型  | 保存的内容        | 使用方式      |
| ----- | ------------ | --------- |
| 情景记忆  | “上次发生了什么”    | 阅读案例后重新推理 |
| 程序性记忆 | “遇到这种情况该怎么做” | 直接调用操作流程  |

情景记忆偏向“查历史”，程序性记忆偏向“直接执行技能”。

## 二、研究方法

ProcMEM主要由三部分组成：

1. Skill-MDP：定义什么是技能，以及如何调用技能；

2. Non-Parametric PPO：从经验中生成和验证新技能；

3. 在线评分机制：保留好技能，删除差技能

**Skill-MDP：把经验表示成技能**

作者在传统马尔可夫决策过程 MDP 中增加了一个技能池，形成 **Skill-MDP**。

智能体的决策分为两层：

1. 先根据当前状态，从技能池中选择一个技能；
2. 再由冻结的 LLM 根据技能说明生成具体动作。

也就是说，LLM不再完全从零开始推理，而是在技能的指导下行动。

一个技能包含三部分

| 组成部分                  | 含义   | 回答的问题   |
| --------------------- | ---- | ------- |
| Activation Condition  | 激活条件 | 什么时候使用？ |
| Execution Procedure   | 执行流程 | 具体怎么做？  |
| Termination Condition | 终止条件 | 什么时候停止？ |

作者测试了两类选择机制：

- **基于相似度**：选择激活条件与当前状态最相似的技能；
- **基于价值**：先找到若干相似技能，再选择预期回报最高的技能。

论文的重点不是训练检索器，而是让技能池本身不断进化。

**Semantic Gradient：用自然语言表示“梯度”**

标准神经网络通过数值梯度更新参数，但ProcMEM不更新参数，因此作者提出了 **Semantic Gradient，语义梯度**。

它本质上是：

> 根据成功或失败的交互轨迹，用自然语言指出某个技能应该如何修改。

**第一步：逐条轨迹进行事后归因**

对于使用了某项技能的轨迹，系统分析：

- 技能是否在错误的时机被激活；
- 执行步骤是否缺失或顺序不合理；
- 技能是否结束得太早或太晚；
- 哪一步导致了最终成功或失败。

然后分别针对技能的三个部分生成修改建议。

例如：

- 激活条件过于宽泛，应增加“目标位置未知”这一限制；
- 执行过程缺少验证步骤；
- 找到目标后应立即终止，而不是继续探索。

**第二步：批量聚合**

单条轨迹可能存在偶然性，因此系统不会根据一次失败立即修改技能。

它会把一批轨迹中的语义梯度汇总，提取：

- 反复出现的失败模式；
- 多条轨迹一致支持的修改方向；
- 可以泛化的改进建议。

同时过滤相互冲突或只适用于个别样本的建议。

**第三步：生成候选技能**

LLM根据聚合后的语义梯度，修改原技能的：

- 激活条件；
- 执行步骤；
- 终止条件。

由此生成若干候选版本。

**PPO Gate：验证候选技能是否可靠**

仅依赖LLM修改技能存在风险：

- 可能产生幻觉；
- 可能把技能改得过于激进；
- 新技能在看似合理，实际执行效果却更差。

因此作者借鉴标准PPO中的“信任区域”思想，设计了 **PPO Gate**。

系统拿历史轨迹进行反事实评估：

> 如果当时使用新技能，而不是旧技能，历史上的高回报动作是否会变得更可能出现？

如果新技能更倾向于支持那些带来高回报的动作，并且与旧技能的行为差异没有过大，就会获得较高分数。

**Best-of-N选择**

系统一次生成多个候选技能：

1. 分别计算PPO Gate分数；
2. 选择分数最高的候选；
3. 只有分数大于0时，才允许它进入技能池。

因此，PPO Gate并不是训练模型，而是一个**技能准入审查机制**。

**在线评分与技能池维护**

技能池容量是有限的，因此不能无限保存所有技能。

系统为每个技能维护一个在线分数，主要衡量：

> 这个技能被调用时，平均带来了多少超过基线的回报。

系统会删除：

- 分数小于或等于0的技能；
- 内容重复或高度相似的技能；
- 当技能池超出容量时，分数最低的技能。

llm：分析轨迹、生成语义梯度、汇总建议、改写技能

### 工作流程

- **智能体与环境交互**  
  冻结参数的 LLM 根据当前状态选择一个已有技能，并执行具体动作，形成完整交互轨迹。
- **分析交互经验**  
  系统回看成功或失败轨迹，判断问题来自技能的激活条件、执行步骤，还是终止条件。
- **生成候选技能**  
  将多条轨迹中的改进建议汇总为“语义梯度”，再利用 LLM 修改旧技能或生成新技能。
- **PPO Gate 验证**  
  用历史轨迹评估候选技能：它是否更支持高收益动作，同时又不会偏离原有行为太多。只有验证分数为正的候选技能才会进入技能库。
- **技能库维护与复用**  
  根据技能实际带来的收益进行在线评分，删除低质量、过时或重复技能；保留下来的技能会在后续相似任务中直接调用。
