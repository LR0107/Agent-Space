# Get Experience from Practice: LLM Agents with Record & Replay

**Arxiv 2025.5 IPADS**

让 AI 智能体不要每次执行任务都“从零思考”，而是把过去成功完成任务的过程保存为经验，以后遇到类似任务时直接参考和复用。

## 一、研究背景

现在的 LLM 智能体可以操作网页、填写表格、调用 API，甚至自主完成多步骤任务。但论文认为，这类智能体仍面临四个主要问题：

| 问题      | 解释                         |
| ------- | -------------------------- |
| **可靠性** | 大模型存在幻觉，同一个任务多执行几次，结果可能不一样 |
| **隐私**  | 截图、文件、操作记录等敏感信息可能需要上传云端    |
| **成本**  | 每一步都调用大模型，尤其是视觉模型，费用较高     |
| **速度**  | 智能体每一步都要观察、推理、规划，通常比人工操作慢  |

仅靠提示词、模型蒸馏或部署小模型，通常只能缓解其中一部分问题，很难同时兼顾可靠性、隐私、成本和效率。

---

## 二、研究方法

完成一次任务 → 记录操作过程 Record → 总结为可复用经验 Summary → 在类似任务中回放 Replay

- **记录 Record**
  保存用户或智能体完成任务时的操作，包括点击、输入、API 调用、页面状态和执行结果。
- **总结 Summary**  
  不只是保存原始操作录像，而是从操作轨迹中提取通用的流程、变量和约束，形成结构化的“经验”。
- **回放 Replay**  
  以后遇到类似任务时，智能体优先按照已有经验执行，不再完全依靠大模型重新规划。

**和传统录制回放的区别**

传统录制回放工具，例如网页自动化脚本，往往会机械地重复。

这种方法速度快、结果稳定，但网页布局稍微变化就可能失效。

### 关键设计

**1.低层级经验**

低层级经验非常具体，例如：

```
点击城市输入框
输入“上海”
```

**优点**

- 执行速度快；
- 对大模型能力要求低；
- 行为较稳定；
- 适合页面和环境没有明显变化的情况。

**缺点**

- 泛化能力弱；
- 页面布局、按钮位置或接口变化后容易失效。

**2.高层级经验**

高层级经验更抽象，例如订酒店的流程可以总结为：

```
1. 选择城市
2. 输入酒店名称或位置
3. 选择入住和退房日期
4. 选择房间及住客人数
```

它不规定必须点击哪个坐标，而是告诉智能体“要完成哪些步骤”。

**优点**

- 能适应不同网站、应用和界面；
- 可以处理任务参数变化；
- 泛化能力较强。

**缺点**

- 仍需要模型根据当前环境生成具体动作；
- 模型能力不足时可能理解错误；
- 执行成本比低层级经验高。

论文并没有给出严格的自动分类公式，而是通过**粒度**区分。

### 工作流程

**1. Record：记录模块**

记录：

- 当前网页或应用状态；
- 页面元素；
- 用户点击、输入、滚动等操作；
- API 调用；
- 每次操作前后的状态变化。

最终得到一条完整的执行轨迹：

```
状态 S1 → 动作 A1 → 状态 S2 → 动作 A2 → 状态 S3
```

**2.Summary：总结模块**

将一次或多次任务轨迹进行合并和抽象：

- 找出不同轨迹中的共同步骤；
- 将变化的内容标记为变量；
- 提取高层和低层经验；
- 生成对应的检查函数。

例如，多次填写表格时，姓名、日期和地点不同，但填写流程相似。系统会保留流程，把具体姓名、日期和地点抽象成参数。

**3.Experience Store：经验库**

经验可以保存在一个类似“应用商店”的经验仓库中，包括：

- 任务说明；
- 经验内容；
- 适用环境；
- 版本；
- 成功率；
- 用户评分；
- 是否经过审核；

用户或智能体可搜索、下载和共享经验。

系统需要先查询并选择“最合适的经验”，而不是全部注入模型。

**4. Replay：回放模块**

接到新任务后，系统会：

1. 搜索与当前任务最相似的经验；
2. 选择合适的经验层级；
3. 根据当前环境生成具体操作；
4. 执行动作；
5. 每一步调用检查函数；
6. 若低层经验失效，则退回高层经验重新规划；
7. 出现危险操作时立即阻止。

## 三、总结

### 优势

**1. 提高可靠性**

智能体不再随意探索，而是在成功经验和检查函数限制下执行，可以减少幻觉导致的错误。

**2.降低调用成本**

很多步骤可以直接回放，不需要每一步都调用大型模型。

**3.提高执行速度**

低层经验接近自动化脚本，能够直接执行具体动作，省去反复推理的时间。

**4.改善隐私**

复杂规划可以预先完成，实际回放时在本地运行，只处理少量必要数据，减少原始截图、文档和个人信息上传云端的需求。

**5.让小模型获得“大模型经验”**

小模型不必具备完整的自主规划能力，只需理解和执行已有经验，因此资源受限设备也可能完成较复杂任务。

### 局限

**1. 如何正确总结经验**

从若干次操作中提取通用经验并不容易：

- 可能漏掉关键步骤；
- 可能错误地合并不同状态；
- 自动生成的脚本可能无法执行；
- 高层和低层经验还缺少统一、清晰的定义。

**2.泛化能力与安全性存在矛盾**

经验太具体：

- 安全稳定；
- 但稍微换个页面就不能用了。

经验太抽象：

- 能适应新环境；
- 但模型自由度增加，风险也增加。

论文目前采用较保守的泛化方式，主要适用于流程清晰、重复性较强的任务，如填表、订酒店和网络购物。

**3.用户需要先付出录制成本**

用户必须至少演示一次或多次任务。共享经验还需要：

- 审核；
- 评分；
- 安全验证；
- 版本维护。

否则，经验商店中的恶意或低质量经验可能造成安全问题。

**4.适用范围有限**

它更适合：

- 重复性强；
- 流程边界明确；
- 输入会变化，但总体步骤相似；
- 可以明确检查成功与否的任务。

它不适合：

- 高度开放的创作；
- 完全陌生的问题；
- 每次任务路径都不同的复杂研究；
- 无法定义明确检查条件的任务。
