# Agent Workflow Memory

**ICML 2025**

如何让基于大语言模型的智能体，从过去完成任务的经验中提炼出可复用的操作流程，并在之后的任务中继续使用这些流程。

## 一、研究背景

基于大语言模型的智能体已经可以完成一些网页任务

但现有智能体在处理**步骤较多、过程较长、任务较复杂**的网页操作时，表现仍然不稳定。

核心问题有三个：

**（1）每个任务都“从头开始”**

传统智能体通常将每个任务视为一个独立问题。

例如，它已经完成过：   

   在地图上搜索“卡内基梅隆大学”。

之后又遇到：查询“卡内基梅隆大学”的邮政编码。

传统智能体往往不会主动复用第一次任务中“搜索地点”的操作经验，而是重新规划整个过程。

这意味着智能体虽然完成了很多任务，却没有真正形成持续积累的能力。

**（2）记住完整案例，但不会抽象规律**

一些已有方法会把过去的完整任务案例放入提示词中，让模型参考。

但是完整案例通常包含大量具体信息，例如：

- 特定商品名称；
- 特定网页元素编号；
- 特定网站；
- 特定任务目标。

这些案例与新任务未必完全匹配，因此容易让模型机械模仿，而不是学习通用规律。

例如：

> 搜索“dry cat food”

更有价值的抽象形式应当是：

> 搜索 `{product-name}`。

论文认为，智能体真正需要学习的不是某一个具体案例，而是案例中可以反复使用的**子流程**。

**（3）难以适应任务和环境变化**

模型可能对与示例高度相似的任务表现较好，但当测试任务来自：

- 不同任务模板；
- 不同网站；
- 不同业务领域；

性能往往明显下降。

因此，论文希望构建一种类似人类学习经验的机制：

> 人不会记住每一次操作的所有细节，而会提炼出“搜索商品”“填写表单”“查询路线”等通用流程，并在新情境中灵活复用。

## 二、研究方法

从成功的智能体操作轨迹中，提炼出可复用的工作流，将其保存到记忆中，再用这些工作流指导后续任务。

**基本任务建模**

论文将网页智能体的工作过程描述为一个反复进行的“观察—行动”循环。

在每一步中：

1. 智能体接收用户任务；
2. 查看当前网页状态；
3. 大语言模型根据任务、网页观察和记忆生成操作；
4. 操作改变网页状态；
5. 智能体继续观察并执行下一步；
6. 直到任务完成或达到最大步骤数。

一个完整任务会形成一条**经验轨迹**，包括：

- 用户指令；
- 每一步的网页观察；
- 智能体的推理；
- 实际执行的操作。

AWM 从这些经验轨迹中提取工作流。

轨迹来源于跑WebArena和Mind2Web的结果

### 工作流的表示方式

每个工作流包括两个主要部分。

（1）工作流描述

用自然语言说明该工作流的目标，例如：

> 查找具有指定编号的客户订单。

这相当于告诉智能体：这套操作是用来解决什么问题的。

（2）工作流轨迹

工作流中包含多个步骤。每一步又包括三部分：

- **环境状态描述**：当前网页处于什么状态；
- **推理说明**：为什么要执行下一步操作；
- **具体动作**：点击、输入、滚动、停止等。

执行任务 → 判断是否成功 → 从成功轨迹中提取工作流 → 将工作流加入记忆。

### 工作流提取

作者主要使用大语言模型作为**工作流归纳模块**。

模型接收一组任务经验，然后寻找其中反复出现的子流程。

作者在提示词中提出了几个关键要求：

**要求一：提取子流程，而不是复制完整任务**

例如，不提取：

> 在亚马逊购买猫粮并寄送到家庭地址。

而是提取：

> 在亚马逊搜索商品。

后者更容易用于其他任务。

**要求二：抽象具体变量**

将：

```
搜索 dry cat food
```

抽象为：

```
搜索 {product-name}
```

将特定地点、订单号、按钮编号等都替换成变量。

**要求三：避免重复或高度重叠的工作流**

如果多个流程本质相同，只保留较通用的版本，减少记忆冗余。

**将工作流加入智能体记忆**

工作流提取完成后，会被作为辅助信息加入智能体的提示上下文。

原来的智能体决策依据是：

```
用户任务 + 当前网页状态 + 基础操作说明
```

加入 AWM 后变成：

```
用户任务 + 当前网页状态 + 基础操作说明 + 已学习的工作流
```

智能体并不是直接调用一个固定程序，而是把工作流当作操作指南，根据当前网页情况决定是否采用、修改或跳过其中的步骤。

### 两种工作模式

AWM 支持两种使用场景：**离线模式**和**在线模式**。

**1.离线 AWM**

离线模式适用于有高质量训练样例的情况。

流程如下：

1. 收集训练集中的成功任务轨迹；
2. 在测试前统一提取工作流；
3. 将工作流加入智能体记忆；
4. 使用固定的工作流集合完成所有测试任务。

可以理解为：

> 先集中学习经验，再参加考试。

这种模式使用的工作流在测试过程中一般不再变化

**2. 在线 AWM**

在线模式不需要额外训练样例，只利用测试过程中已经完成的任务。

流程是：

1. 智能体尝试完成当前任务；
2. 评估模块判断任务是否完成正确；
3. 如果任务成功，就从这次轨迹中提取工作流；
4. 将新工作流加入记忆；
5. 使用更新后的记忆完成下一个任务。

可以理解为：

> 边做任务，边总结经验，边改进。

论文对比了两种模式：

- 离线模式：先提取，再统一使用；
- 在线模式：每完成一个成功任务，就继续扩展记忆。

在线模式的优势在于，它提取的工作流直接来自当前测试环境，因此能够适应新的任务、网站和领域。

不过它也有一个风险：如果评估模块把错误轨迹误判为成功，就可能学到错误工作流。

---

### 工作流程

- **执行任务**  
  智能体根据用户指令观察网页，并连续执行点击、输入、滚动等操作，形成一条完整的操作轨迹。
- **判断是否成功**  
  任务完成后，由评估模块判断这条轨迹是否真正解决了问题。失败轨迹不会进入记忆。
- **提取工作流**  
  从成功轨迹中抽取可重复使用的子流程，并将具体内容抽象成变量。  
  例如，把“搜索北京大学”总结为“搜索 `{location}`”。
- **写入并复用记忆**  
  将提取出的工作流加入智能体记忆（直接拼进模型上下文中，未涉及到检索）。之后遇到类似或更复杂的任务时，智能体可以直接参考这些流程，而不是从头规划。

## 三、实验设计

### 实验设置

数据集：

1. **WebArena**：包含812个网页导航任务，覆盖5类网站环境，涉及4种常见应用领域
2. **Mind2Web**

实验总体覆盖了超过1000个任务、200多个网站或领域环境。

**Baseline**

BrowserGym：一种较强的自主网页智能体基线，优化了动作空间和网页操作框架。

AutoEval：通过额外的自动评估和修正步骤，反复改进任务轨迹。

SteP：使用人类专家提前编写的14个网站专用工作流。这一比较特别重要，因为 SteP 使用了人工编写流程，而 AWM 的工作流是智能体自动从经验中提取的。

WebArena 的评价指标

**（1）任务成功率**：最终正确完成任务的比例。

（2）**平均操作步骤数：**成功完成任务所需的平均步骤数。

后者用于评估智能体效率：成功率相同的情况下，步骤越少越好。

### 实验结果

WebArena 结果

| 方法                | 总任务成功率    | 平均步骤数   |
| ----------------- | --------- | ------- |
| WebArena 原始方法     | 14.9%     | —       |
| AutoEval          | 20.2%     | 46.7    |
| BrowserGym        | 23.5%     | —       |
| BrowserGym（仅无障碍树） | 15.0%     | 7.9     |
| SteP（人工工作流）       | 33.0%     | —       |
| **AWM**           | **35.5%** | **5.9** |

**少量样本学习实验**

作者进一步分析了在线 AWM 随任务数量增加的学习过程。

- 在前约40个任务中，成功率快速上升；
- 此时智能体主要学会一些基础、常用工作流；
- 40个任务之后，性能逐渐稳定；
- 后续任务主要用于积累更复杂的工作流。

这说明 AWM 不需要几百或几千个案例，仅通过几十次成功经验就可以获得明显收益。

**跨模板泛化实验**

WebArena 中部分任务来自同一个模板，只是替换了商品、地点或用户名等参数。

为了避免结果只是因为模型“记住了同模板任务”，作者设计了一个更严格的子集：

1. 按任务模板分组；
2. 每个模板只随机保留一个样例；
3. 确保测试任务之间不存在模板重复；
4. 比较不同方法的成功率。

结果如下：

| 方法         | 跨模板任务成功率  |
| ---------- | --------- |
| BrowserGym | 20.5%     |
| AutoEval   | 23.2%     |
| SteP       | 32.1%     |
| **AWM**    | **33.2%** |

这表明 AWM 学到的并不只是某个固定模板，而是能够迁移到不同任务中的通用流程。

### 消融实验

**大语言模型归纳与规则归纳**

作者设计了一个规则方法：

- 提取任务的动作序列；
- 根据动作类型去重；
- 删除无法执行的步骤；
- 将剩余完整轨迹当作工作流。

在 WebArena 中：

| 方法    | 成功率  | 平均步骤    |
| ----- | ---- | ------- |
| 规则归纳  | 35.6 | 6.3     |
| LM 归纳 | 35.5 | **5.9** |

二者成功率非常接近，但 LM 工作流步骤更精炼。

## 四 、总结

工作流直接注入上下文，会导致爆炸，没涉及到检索

失败轨迹直接丢弃，可以用来反思/指导

轨迹生成要自己跑 成本高 耗时久 可以考虑自动生成

只在网页任务中跑 可迁移到其他场景

增加归纳、评估，对总token成本分析少

---

### 创新点

把“完整经验”压缩成“可复用工作流”：过去很多方法直接把历史完整轨迹放进上下文，问题是这种高度依赖具体任务，迁移性弱。

支持在线、无监督地持续积累记忆：AWM 不要求一定有人工标注轨迹。

在不修改模型参数的情况下提升长期任务能力

### 不足

**没有真正的工作流检索机制：**当工作流数量较少时问题不大，但长期运行后会出现：

- 上下文越来越长；
- 无关工作流干扰决策；
- token 成本增加；
- 模型难以判断该用哪一个流程。

**失败轨迹没有被利用：**在线 AWM 中，失败轨迹基本直接丢弃。

但失败其实可以提供很多有价值的信息，例如：

- 哪个步骤容易出错；

- 哪些动作会导致死循环；
- 某个流程适用边界是什么。

论文没有利用失败轨迹做：反思；错误归因；正负样本对比。

因此它只学习“应该怎么做”，没有系统学习“什么情况下不应该这样做”。

**高度依赖成功评估器的准确性：**只有成功轨迹才能进入记忆。但这里的成功通常由模型评估器判断，不一定完全可靠。

缺少记忆维护与遗忘机制

论文关注的是“如何增加工作流”，但没有讨论“如何管理工作流”。

没有系统解决：

- 重复工作流合并；
- 冲突工作流消解；
- 低质量工作流淘汰；
- 过时流程更新；

### Q&A

**1.工作流具体包含什么内容？**

每个工作流通常包含两部分：第一部分是自然语言描述，说明这个流程解决什么问题；第二部分是操作轨迹，包括当前环境状态描述、智能体的推理以及点击、输入、滚动等动作。具体任务中的商品名、地点名和订单号会被替换成变量，以提高泛化能力。

**2.工作流是怎么生成的？**

作者主要使用大语言模型作为工作流归纳器。模型读取一条或多条成功轨迹，找出其中重复出现的子流程，将任务特定内容替换为变量，并避免产生相似或重叠的流程。论文还比较了规则归纳方法，语言模型归纳效果更好。

**3.在线模式怎么判断一条轨迹是否成功？**

论文使用一个语言模型评估模块，对用户任务和完整操作轨迹进行判断，输出成功或失败的二元标签。只有被判断为成功的轨迹才会用于归纳工作流。这个设计简单，但评估器如果误判，也可能把错误经验写入记忆。

**4.这套方法的计算成本如何？**

AWM可以减少任务执行步骤，但会增加额外的大模型调用，包括成功评估和工作流归纳。随着工作流数量增加，动作生成时的输入token也会上升。论文主要报告了执行步骤效率，对总token成本、模型调用成本和长期内存增长的分析还不充分。
