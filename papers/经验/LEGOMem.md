# LEGOMem: Modular Procedural Memory for Multi-agent LLM Systems for Workflow Automation

**AAMAS 2026**

LEGOMem 会将过去成功完成任务的过程整理成可复用的记忆，并在遇到新任务时，把不同粒度的记忆分配给负责规划的协调器和负责具体操作的任务智能体。

## 一、研究背景

办公自动化任务往往涉及多个应用。例如：

> 从日历中提取会议安排 → 整理信息 → 创建 Excel 文件 → 将会议内容写入表格。

一个单一智能体需要同时掌握日历、邮件、Excel、Word 等多种工具，系统会变得复杂。因此，当前常见做法是采用多智能体架构：

- 一个中央协调器负责规划和任务分解；
- 多个专业智能体分别操作 Calendar、Email、Excel、Word 等工具。

这种结构具有较好的模块化和可扩展性。

### 现有系统有什么不足

论文认为，当前多智能体系统普遍存在一个关键缺陷：**无状态、缺少长期经验积累**。

每次收到任务后，系统都会：

1. 重新分析任务；
2. 重新制定计划；
3. 重新探索工具调用方式；
4. 可能重复以前犯过的错误。

任务执行完成后，成功经验并不会被保留下来。

已有的记忆研究主要面向单智能体系统，例如：

- Synapse：保存完整的历史执行轨迹；
- AWM：从历史轨迹中提取可重复使用的子任务流程。

但多智能体系统不仅要考虑“记什么”，还要考虑：

- 记忆应该给协调器，还是给任务智能体？
- 不同智能体应该得到哪些记忆？
- 应该在什么时候检索记忆？
- 是检索完整任务，还是检索具体子任务？

## 二、研究方法

程序性记忆不是记住某个事实，而是记住：

> “完成某类任务时，应该按什么流程做。”

程序性记忆更像是可复用的操作流程、工具调用经验和任务解决策略。

**两类记忆**

LEGOMem 将**成功任务**的执行轨迹拆成两种记忆。

**① 完整任务记忆**

完整任务记忆主要提供给协调器，包括：

- 原始任务描述；
- 高层任务计划；
- 子任务分解方式；
- 各子任务分配给哪个智能体；
- 最终答案；
- 对执行过程的简短总结或反思。

它帮助协调器回答：

> 这种任务通常应该怎样拆解？  
> 应该先调用哪个智能体？  
> 任务停滞后应该怎样重新规划？

**② 子任务记忆**

子任务记忆主要提供给具体任务智能体，包括：

- 子任务描述；
- 负责该子任务的智能体类型；
- 智能体的推理过程；
- 工具调用动作；
- 环境返回结果；
- 子任务执行结果。

它帮助任务智能体回答：

> 这个工具具体怎么调用？  
> 参数应该怎样填写？  
> 操作步骤应该是什么？

### 两阶段方法

LEGOMem 的整体方法分为两个阶段。

**阶段一：离线构建记忆库**

这一阶段类似于“整理学习资料”。

研究者首先让多智能体系统在没有记忆的情况下完成一批任务，然后：

1. 收集执行日志；
2. 筛选**成功完成**的任务；
3. 提取协调器的规划过程；
4. 提取各任务智能体的工具操作轨迹；
5. 使用大模型将原始日志整理成结构化记忆；
6. 将记忆转换成向量，存入向量数据库。

只有成功轨迹被用来构建主要记忆库。

论文最终从训练任务中提取了：

- 93 条完整任务记忆；
- 250 条子任务记忆。

使用一个 LLM，把这些较长、较杂乱的执行日志转换成结构化的记忆单元。

任务本身来自 OfficeBench；执行轨迹由作者运行其 GPT-4o 多智能体系统自动产生；成功与否由程序化评测确定；随后再由 LLM 自动整理成记忆。

**阶段二：在线检索并分配记忆**

当新任务到来时，系统会：

1. 将新任务描述转换为语义向量；
2. 从记忆库中检索最相似的历史任务；
3. 将完整任务记忆交给协调器；
4. 将相关子任务记忆交给对应任务智能体；
5. 协调器结合当前状态和历史经验制定计划；
6. 各任务智能体参考子任务记忆调用工具；
7. 如果执行停滞，协调器利用记忆重新规划。

系统为协调器检索 5 条记忆，为每个任务智能体检索 3 条记忆。

### 三种 LEGOMem 变体

论文设计了三种记忆检索方式，用于比较“记忆检索粒度”是否会影响系统表现。

**1. Vanilla LEGOMem**

最简单的版本。

处理方式是：

1. 根据整个新任务检索相似的完整任务；
2. 将完整任务记忆提供给协调器；
3. 从这些完整记忆中提取子任务部分；
4. 按照智能体类型分配给各任务智能体。

**优点**

- 结构简单；
- 检索次数少；
- 推理速度较快；
- 总体表现稳定。

**局限**

两个任务整体上可能相似，但其中某个子任务并不相似。因此，从完整任务中直接拆出来的子任务记忆，不一定是当前智能体最需要的。

**2. LEGOMem-Dynamic**

Dynamic 表示“动态检索”。

系统为每类任务智能体建立独立记忆库，例如：

- Calendar 智能体记忆库；
- Email 智能体记忆库；
- Excel 智能体记忆库。

协调器生成一个具体子任务后，系统才根据该子任务描述，在对应智能体的记忆库中进行检索。

例如，协调器生成：

> “读取 Bob 收件箱中所有邮件，并比较发送时间。”

系统便会实时从 Email 智能体记忆库中检索最相关的邮件操作经验。

**优点**

- 子任务记忆更加精确；
- 无关记忆较少；
- 对能力较弱的小模型更有帮助。

**局限**

每执行一个新子任务，都要重新进行向量计算和检索，运行成本较高。

**3. LEGOMem-QueryRewrite**

这一版本将子任务检索提前到规划阶段。

流程是：

1. 先检索相似的完整任务记忆；
2. 由一个“查询改写模型”生成新任务的初步子任务计划；
3. 用这些改写后的子任务描述检索各智能体的记忆；
4. 在正式执行前，把记忆提前分配好。

它试图同时获得：

- Dynamic 版本的细粒度检索能力；
- 比 Dynamic 更少的运行时检索开销。

### 工作流程

**1. 离线构建记忆库**

先让多智能体系统在**不使用记忆**的情况下完成一批办公任务，然后：

1. 收集成功任务的执行轨迹；
2. 提取协调器的任务规划和分工过程；
3. 提取各任务智能体的工具调用步骤；
4. 整理成两类记忆：
   - **完整任务记忆**：保存整体计划和任务分解；
   - **子任务记忆**：保存某个智能体的具体操作步骤；
5. 将这些记忆存入向量数据库。

**2. 在线执行新任务**

收到新任务后：

1. 根据任务描述检索相似的历史记忆；
2. 将完整任务记忆提供给协调器；
3. 协调器制定计划，将任务拆成多个子任务；
4. 将相关子任务记忆分配给 Calendar、Email、Excel 等专业智能体；
5. 各智能体参考过去的操作经验调用工具；
6. 执行结果返回协调器，协调器继续安排下一步；
7. 如果任务停滞，协调器利用记忆重新规划，直到完成任务。

## 三、实验设计

### 实验设置

**数据集**

使用 **OfficeBench** 作为评测基准，包含 300 个办公自动化任务，将其划分为：

| 数据用途       | 任务数量 |
| ---------- | ---- |
| 训练集，用于构建记忆 | 148  |
| 测试集，用于评价效果 | 152  |

任务被分为三个难度等级：

- **Level 1**：只涉及一个应用；
- **Level 2**：涉及两个应用；
- **Level 3**：涉及多个应用的复杂工作流。

评价指标是**任务成功率**，即最终环境状态是否符合预期，例如：

- 邮件是否正确发送；
- 问答结果是否匹配。

**多智能体组成**

研究者设计了三种团队配置。

| 团队     | 协调器         | 任务智能体       |
| ------ | ----------- | ----------- |
| LLM 团队 | GPT-4o      | GPT-4o      |
| 混合团队   | GPT-4o      | GPT-4o-mini |
| SLM 团队 | GPT-4o-mini | GPT-4o-mini |

**对比方法**

| 方法                   | 说明              |
| -------------------- | --------------- |
| No Memory            | 不使用任何历史记忆       |
| Synapse              | 直接使用相似任务的完整原始轨迹 |
| AWM                  | 从历史轨迹中总结可复用子任务  |
| LEGOMem              | 模块化完整任务记忆与子任务记忆 |
| LEGOMem-Dynamic      | 执行时动态检索子任务记忆    |
| LEGOMem-QueryRewrite | 规划阶段改写子任务并提前检索  |

为了公平比较，Synapse 和 AWM 也使用相同的 93 条成功轨迹进行记忆构建。

**消融实验**

除了比较总体成功率，论文还进行了多组消融实验。

**① 记忆放在哪里？**

比较：

- 协调器和任务智能体都有记忆；
- 只有协调器有记忆；
- 只有任务智能体有记忆；
- 协调器只在规划阶段使用记忆；
- 完全没有记忆。

目的在于确认：

> 高层规划记忆和底层执行记忆，哪个更重要？

**② 检索方式是否重要？**

比较：

- Vanilla；
- Dynamic；
- QueryRewrite。

目的是判断细粒度子任务检索是否比简单的整体任务检索更有效。

**③ 记忆中是否需要保存推理过程？**

研究者比较了：

- 记忆中包含轻量推理信息；
- 只保留结构化任务步骤，不包含推理。

目的是判断“思考过程”是否是记忆发挥作用的必要条件。

**④ 是否减少执行步骤与错误？**

研究者还统计了：

- 平均工具调用步数；
- 错误工具调用的比例。

这可以评价 LEGOMem 是否不仅“成功率更高”，而且“执行得更快、更稳定”。

### 实验结果

与无记忆系统相比，Vanilla LEGOMem 的总体成功率提升为：

| 团队     | 无记忆    | LEGOMem | 提升     |
| ------ | ------ | ------- | ------ |
| LLM 团队 | 45.83% | 58.44%  | +12.61 |
| 混合团队   | 35.31% | 48.03%  | +12.72 |
| SLM 团队 | 24.78% | 38.16%  | +13.38 |

消融实验表明：

- 同时给协调器和任务智能体提供记忆，效果最好；
- 只给协调器记忆，仍然能获得明显提升；
- 只给任务智能体记忆，效果较弱；
- 如果没有全局规划记忆，即使局部工具操作更准确，整体任务也可能失败。

全局规划错误通常比某一步工具操作错误影响更大。

**细粒度检索更适合小模型智能体：**小模型更依赖直接、具体、与当前子任务高度相关的操作示例

**保存额外推理过程作用有限：**在记忆中加入推理过程后，总体分数变化通常不到 2 个百分点，有时甚至略有下降。

**能减少执行步数和错误**

在 Level 3 复杂任务中：

- 无记忆系统平均需要 26.5 步；
- 完整 LEGOMem 平均需要 22.2 步；
- 执行步数减少约 **16.2%**。

错误步骤率从：0.275 降至 0.225。

## 四、总结

### 创新点

**把“程序性记忆”从单智能体扩展到多智能体，并且按角色拆分**

以往不少记忆方法主要面向单智能体。

多智能体系统中，协调器和执行智能体需要的知识并不相同：协调器需要知道“任务应该怎么拆”，任务智能体需要知道“这一步具体怎么做”。

## 不足

**主要使用成功轨迹构建记忆，因此失败经验并未被充分利用。**

- 148 个训练任务；
- 得到 93 条成功完整轨迹；
- 再提取出 250 条子任务记忆。

**任务规模和记忆库规模较小**

**成本和效率分析不充分**

论文报告了执行步数和错误率下降，但没有系统比较：LLM token 消耗和向量检索开销。

### Q&A

**1.程序性记忆与普通的语义记忆、情景记忆有什么不同？**

语义记忆主要保存事实知识，情景记忆主要保存发生过的具体事件；程序性记忆关注的是“任务应该怎么完成”。

**2.为什么要把记忆分成完整任务记忆和子任务记忆？**

因为协调器和任务智能体承担的职责不同。协调器需要了解任务整体结构、执行顺序以及该选择哪个智能体；任务智能体更需要具体的工具使用方法和局部执行步骤。如果只保存完整轨迹，信息会比较冗余；如果只保存局部步骤，又会缺乏全局协调

**3.实验为什么设计三种不同的模型团队？**

这是为了研究记忆是否只对大模型有效。三种配置分别是全 GPT-4o 团队、GPT-4o 协调器加 GPT-4o-mini 任务智能体的混合团队，以及全部使用 GPT-4o-mini 的小模型团队。结果显示，三种团队都能从记忆中受益，而且小模型团队的相对提升非常明显。

**4.使用记忆会不会增加成本？**

会增加部分成本，包括历史轨迹总结、embedding 计算、向量检索以及将记忆加入上下文所产生的 token 开销。论文主要报告任务成功率、执行步数和错误率，没有提供完整的延迟和经济成本分析。
