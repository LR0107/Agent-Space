# AutoGuide: Automated Generation and Selection of Context-Aware Guidelines for Large Language Model Agents

**NeurIPS 2024**

如何让大语言模型智能体从过去的交互经验中自动总结“操作规则”，并在当前场景下只调用真正相关的规则，从而减少错误决策。

## 一、研究背景

近年来，大语言模型不仅可以回答问题，还可以作为智能体完成连续决策任务，例如：

- 在网站上搜索并购买商品；
- 在 Reddit、GitHub 等平台执行操作；

这类任务不是一次性生成答案，而是一个连续过程：

> 观察环境 → 判断当前状态 → 选择动作 → 获得新观察 → 继续决策。

虽然大语言模型具备一定的推理和规划能力，但当它面对不熟悉、复杂或动态的环境时，往往会犯一些基础操作错误。

例如在网页任务中，模型可能：

- 点击错误的按钮；

- 在错误页面执行操作；

- 忽略前面已经做过的步骤；

- 受到无关提示的干扰。

传统上下文学习难以充分利用历史经验

常见方法是把若干成功操作轨迹作为示例放进提示词中，让模型模仿，这就是基于示例的上下文学习。

但这种方式存在几个问题：

**第一，轨迹太长**

网页操作或机器人任务通常包含很多步骤。直接把大量历史轨迹塞入提示词，很容易超过模型的上下文长度限制。

**第二，示例中包含大量无关信息**

完整轨迹中，真正有用的可能只是某一步的关键经验。模型却必须阅读整条轨迹，信息利用效率较低。

**第三，示例数量增加后，性能提升有限**

更多示例并不一定持续带来收益，反而可能让模型难以判断当前应该参考哪一个示例。

**第四，固定规则可能误导模型**

已有方法 ExpeL 也会从经验中提取自然语言规则，但它通常把所有规则一起提供给智能体。

问题：一条规则可能只适用于某个特定场景。如果在其他场景中也出现，模型可能错误套用。

因此，作者认为关键不只是“提取规则”，而是：

> 必须同时说明规则在什么上下文中适用，并在测试时只选择当前场景真正相关的规则。

## 二、研究方法

AutoGuide 的基本思路可以概括为两步：

1. **从成功与失败的历史轨迹中提取经验；**
2. **根据当前状态，选择与当前场景相关的经验。**

例如，智能体需要在 Reddit 首页进入某个论坛。历史经验中有两条轨迹：

- 成功轨迹：点击 “Forums”；
- 失败轨迹：点击 “Wiki”。

AutoGuide 会把这两条轨迹进行比较，并生成：

- **上下文**：当前位于 Reddit 首页；
- **指南**：当你位于 Reddit 首页并希望进入特定论坛时，应点击位于 “Wiki” 上方的 “Forums” 链接。

测试时，如果智能体再次位于 Reddit 首页，系统就会检索并提供这条指南，帮助模型选择正确动作。

这条指南并不是笼统地说“点击 Forums”，而是采用条件形式：

> 当处于某种场景时，如果目标是什么，应执行什么操作。

这种结构就是论文所说的上下文感知指南。

AutoGuide 使用过去已经收集好的交互轨迹作为离线数据。

作者重点使用同一个任务下的两条对比轨迹：

- τ+：回报较高的轨迹，可以理解为成功轨迹；
- τ−：回报较低的轨迹，可以理解为失败轨迹。

找出成功和失败开始分叉的位置

AutoGuide 会比较同一任务的成功轨迹和失败轨迹，找到它们第一次采取不同动作的时刻。

这个时间点非常重要，因为它通常对应：

> 导致最终成功或失败的关键决策点。

例如，两条轨迹在前面观察到的网页内容完全相同：

- 成功轨迹点击了 Forums；
- 失败轨迹点击了 Wiki。

那么这一步就是分叉点。

AutoGuide 更关注这个关键动作前的状态和动作差异。

### 核心模块

**上下文识别模块**

它的作用是把当前复杂的交互历史压缩成一句简短的自然语言描述。

例如，原始观察可能是：

```
[33] link 'Home'[65] link 'Forums'[66] link 'Wiki'
```

上下文识别模块会将其总结为：

> 当前位于 Reddit 首页。

这样做的意义是将冗长、底层的网页元素或环境状态，转换成更高层、更容易理解的状态表示。

**指南提取模块**

该模块会同时查看：

- 成功轨迹；
- 失败轨迹；
- 当前上下文；
- 两条轨迹在关键步骤采取的不同动作。

然后生成一条自然语言规则。

例如：

> 当位于 Reddit 首页，并希望前往特定论坛时，应点击 “Forums”，而不是 “Wiki”。

**上下文识别模块和指南提取模块都由大模型完成**

**构建“上下文—指南”词典**

AutoGuide 会把所有生成的内容组织成类似词典的数据结构：

```
上下文 A：
    指南 1
    指南 2
```

其中：

- 键是上下文；
- 值是适用于该上下文的指南集合。

为了减少重复，如果两个上下文表达不同但含义相近，系统会使用大语言模型判断它们是否属于同一个场景，并进行合并。

### 测试

在测试任务中，每一步都会执行以下流程：

**第一步：识别当前上下文**

模型根据当前观察和历史动作，总结当前所处状态。

**第二步：查找该上下文下的候选指南**

系统从上下文—指南词典中找到对应指南。

不是传统的向量嵌入或余弦相似度检索，而是使用 LLM 进行语义等价判断。

**第三步：选择最相关的 Top-k 条指南**

如果某个上下文下有很多指南，系统不会全部提供，而是再让模型选出最相关的 k 条。

**第四步：生成动作**

最终智能体根据以下内容做决定：

- 原始任务；
- 当前观察；
- 历史轨迹；
- 当前上下文；
- 选出的相关指南。

AutoGuide 和 ExpeL 都会从经验中生成自然语言规则，但关键区别如下：

| 方法        | 是否使用离线经验 | 指南是否绑定上下文 | 测试时是否筛选指南 |
| --------- | -------- | --------- | --------- |
| ReAct     | 否        | 否         | 否         |
| ExpeL     | 是        | 否         | 通常提供固定规则  |
| AutoGuide | 是        | 是         | 是         |

ExpeL是：即使某条规则与当前状态无关，也可能被放进提示词。

AutoGuide 则强调：

> 规则必须先与上下文绑定，之后再根据当前情境检索。

---

### 工作流程

**1. 训练阶段：从历史经验中生成指南**

- 收集同一任务下的成功轨迹和失败轨迹。
- 比较两条轨迹，找到它们第一次采取不同动作的关键步骤。
- 用上下文识别模块总结当时的状态，例如“当前位于 Reddit 首页”。
- 用指南提取模块分析正确动作和错误动作，生成规则，例如：

> 当位于 Reddit 首页并想进入某个论坛时，应点击 “Forums”，而不是 “Wiki”。

- 将生成的指南按照上下文分类，保存到“上下文—指南”词典中。

**2.测试阶段：动态检索和使用指南**

智能体执行新任务时，每一步都会：

1. 根据当前观察和历史操作识别所处上下文；
2. 从指南词典中查找该上下文对应的规则；
3. 选择最相关的 Top-k 条指南；
4. 将上下文、指南和当前任务一起输入大语言模型；
5. 由模型生成下一步动作，并继续与环境交互。

整体可以概括为：

> **对比成功与失败轨迹 → 定位关键决策点 → 总结上下文 → 提取指南 → 测试时按上下文检索指南 → 辅助智能体决策**

## 四、总结

### 创新点

利用“成功轨迹 vs 失败轨迹”的对比自动总结规则

### 不足

**缺少对指南质量的直接评价标准**

目前判断指南是否有效，主要依靠：把指南用于任务，然后观察最终成功率是否提高。但系统缺少独立的指标来判断。

**多个模块都依赖大语言模型**

上下文识别、规则提取、上下文匹配、指南选择都调用大语言模型，这意味着：推理成本增加；运行速度降低；模型自身错误可能被传递到后续环节。

**依赖高质量、多样化的离线数据：** AutoGuide 的知识来源于历史轨迹。

### Q&A

**1.AutoGuide 和 ExpeL 最大区别是什么？**

两者都尝试从历史经验中提取自然语言知识，但区别在于知识是否和上下文绑定。ExpeL 会生成一些通用规则，然后全部提供给智能体；而 AutoGuide 会给每条规则附加适用条件，并根据当前状态选择相关规则。因此 AutoGuide 可以减少无关知识干扰。

**2.为什么 AutoGuide 比直接增加 few-shot 示例效果更好？**

增加 few-shot 示例本质上仍然是把完整轨迹输入模型，但完整轨迹包含大量冗余信息，而且受到 token 限制。AutoGuide 相当于对历史轨迹进行知识蒸馏，把长轨迹压缩成简洁规则，并且只在需要的时候调用。

**3.AutoGuide 可以和 Reflexion 结合吗？为什么？**

可以。因为两者学习的信息来源不同。Reflexion 主要利用当前任务执行过程中的反馈，属于任务内学习；AutoGuide 利用多个历史任务经验，属于跨任务学习。因此二者具有互补性。
