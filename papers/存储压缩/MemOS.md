# MemOS: A Memory OS for AI System

**Emnlp 2025**

作者认为未来的大模型不仅需要“会推理”，还需要像人一样拥有长期记忆，因此应该像计算机操作系统管理内存一样，专门设计一个记忆操作系统（Memory Operating System）来管理AI的记忆。

## 一.研究背景

虽然 GPT、Claude 等大模型能力越来越强，但它们本质上仍然是：

- 参数里存知识（Parameter Memory）
- 上下文窗口里存短期信息（Context Memory）

导致几个问题：

**（1）长对话容易忘记前文**

因为上下文太长，前面的信息被截断了。

**（2）知识更新困难**

例如：

- 法律条文更新
- 新闻事件变化
- 科学发现出现

模型参数不会自动更新。

解决方法通常是：RAG（检索增强生成）

但作者认为：RAG只是“临时补丁”，不是完整记忆系统。

因为它没有：生命周期管理、版本控制、权限控制

**（3）无法真正个性化**

作者认为：

未来AI应该像人一样长期记住：

- 用户偏好
- 工作习惯

而不是每次重新开始。

**（4）记忆无法跨平台迁移**

在ChatGPT、Cursor、Claude积累了很多经验。

但这些记忆彼此隔离。

作者称之为：Memory Island（记忆孤岛）

---

## 二.研究方法

核心思想：Memory as a System Resource

即：把记忆当成和CPU、内存、硬盘一样的重要系统资源。

| MemOS管理 | 内容        |
| ------- | --------- |
| 参数记忆    | 模型权重      |
| 激活记忆    | KV Cache等 |
| 文本记忆    | 外部知识库     |
| 生命周期    | 创建→使用→归档  |

### 记忆模型

把记忆分成三层。

**第一层：Plaintext Memory（文本记忆）**

例如：

- RAG知识库
- 用户笔记
- 规则文档
- Prompt模板

特点：可编辑、可追踪、易更新

例如：

```
用户喜欢：Python、深色主题
```

直接存成文本。

**第二层：Activation Memory（激活记忆）**

就是模型推理过程中的：

- KV Cache
- Hidden State
- Steering Vector

特点：

- 推理时快速访问
- 短期存在
- 速度快

类似人的：工作记忆

**第三层：Parameter Memory（参数记忆）**

就是：模型权重

例如：

- GPT参数
- LoRA参数

特点：

- 最稳定
- 更新最贵
- 长期保存

类似人的：长期知识

**三种记忆之间可以互相转换**

记忆不是固定的。

可以正向转换，也可以反向转换。

例如：

用户反复提到：我喜欢Python

系统发现使用频率极高，于是：

Plaintext  
↓  
Activation  
↓  
Parameter

最终固化到模型能力中。

---

### MemCube

这是 MemOS 的最小记忆单元。

一个 MemCube 包含：

**1.Memory Payload**

真正内容

例如：

```
{"content": "用户喜欢Python"}
```

**2.Metadata**

元数据

例如：

```
{"created":"2025-04-10","usage":78, "priority":"high", "access":["user","admin"]}
```

包括：创建时间、使用次数、来源、权限

可以理解成：文件系统里的文件 + 元数据

### 架构设计

MemOS采用三层结构。

**第一层：接口层（Interface Layer）**

负责理解用户需求。

核心模块：**MemReader**

作用：把自然语言翻译成记忆操作。

例如：

用户：帮我找去年住院期间医生关于药物的建议

MemReader提取：

```
时间：去年主题：药物任务：检索
```

变成标准查询。

**第二层：操作层（Operation Layer）**

控制中心。

包括：

**MemOperator**

负责：

- 建索引
- 建知识图谱
- 检索记忆

**MemScheduler**

负责：

- 选哪些记忆
- 加载顺序
- 优先级

类似CPU调度器。

**MemLifecycle**

管理生命周期：生成 → 激活 → 合并 → 归档 → 删除

**第三层：基础设施层**

包括：**MemVault**——记忆仓库

**MemGovernance**：记忆治理

负责：权限控制、审计、隐私保护、水印

注：没有公开 MemReader、MemScheduler 的源码实现细节

很多描述停留在：负责什么、输入什么、输出什么

---

### 工作流程

- **解析请求（MemReader）**
  
  - 理解用户意图、时间范围、主题等信息。
  - 将自然语言转换成标准化的记忆查询。

- **检索与组织记忆（MemOperator）**
  
  - 从历史对话、知识库、用户偏好等记忆中找到相关内容。

- **调度记忆（MemScheduler）**
  
  - 决定加载哪些记忆，以及加载顺序和优先级。

- **增强推理**
  
  - 将检索到的记忆注入模型，与当前问题一起进行推理。

- **记忆更新（MemLifecycle）**
  
  - 根据使用情况更新记忆状态，例如激活、合并、归档或删除。

- **存储与治理**
  
  - 保存到 MemVault，并通过 MemGovernance 管理权限、隐私和生命周期。
  
  **MemOS = 解析需求 → 检索记忆 → 调度记忆 → 辅助推理 → 更新记忆 → 存储治理**

例子：

用户输入：“请帮我找一下去年住院期间医生给我的用药建议。”

**1.MemReader：理解需求**

先解析出关键信息：

```
任务：查询时间：去年主题：用药 建议场景：住院期间
```

然后生成标准化的记忆请求。

**2.MemOperator：检索记忆**

根据这些条件，从记忆库中找到：去年的住院记录、医生备注、用药记录、诊断信息

**3.MemScheduler：调度记忆**

从检索结果中挑选最相关的内容：

```
住院记录（高优先级）医生建议（高优先级）无关检查记录（低优先级）
```

只把最有用的记忆送给模型。

**4.LLM生成回答**

模型结合这些记忆后回答：“根据您去年住院期间的记录，医生建议继续服用XX药物，并定期复查……”

这时回答已经利用了历史记忆，而不仅仅是当前提问。

**5.更新记忆**

如果用户后面又确认：“这个建议后来一直有效。”

系统会提高这条记忆的重要性，成为更稳定的长期记忆。

## 三.实验设计

### 实验环境

对比：MIRIX、Mem0、Zep、Memobase、MemU、Supermemory等记忆框架

**测试集**

PreFEval：用户偏好记忆能力

PersonaMem：人格记忆能力

LongMemEval：长程记忆能力

 LoCoMo：复杂长期对话中的记忆和推理能力。

### 实验结果

四类基准上均排名第一。

使其比传统RAG更接近真正长期记忆系统。

---

## 四.总结

MemOS并不是提出一个新的大模型，而是提出一种“AI记忆操作系统”架构：把文本记忆、激活记忆和参数记忆统一管理，通过 MemCube、调度器和生命周期机制，让未来的 AI 像拥有长期记忆的人一样持续学习、个性化成长和跨平台迁移。

### 创新点

**1.提出“Memory OS（记忆操作系统）”框架**

在 MemOS 之前，大部分工作**把记忆当成一个模块（Module）**

而 MemOS 提出：**把记忆当成一种系统资源（System Resource）**

**2.统一三种记忆**

MemOS第一次明确提出：Plaintext Memory、Activation Memory、Parameter Memory

三者应该放在同一个框架里管理。

并且允许动态转换

### 不足

1.更像系统设计论文，而不是算法论文：MemScheduler没有算法实现细节

2.很多模块还停留在概念层：例如生命周期什么时候 Merged？怎么判断 Expired？阈值是多少？

## Q&A

**1.MemOS和RAG有什么本质区别？**

RAG本质上是一种“检索增强生成”方法，其流程是“检索→拼接Prompt→生成回答”。检索到的信息只是临时参与当前推理，不具备生命周期管理能力。

MemOS则把记忆视为系统资源，不仅支持检索，还支持记忆的创建、更新、融合、归档和删除。它关注的是整个记忆生命周期管理，而不仅仅是检索阶段。因此作者认为MemOS是一个记忆操作系统，而RAG只是一个记忆调用工具。

**2.为什么需要三种记忆？**

文本记忆更新方便但访问成本较高；参数记忆推理效率高但更新成本巨大；激活记忆响应速度快但生命周期较短。

作者希望利用三种记忆互补，实现效率与灵活性的平衡。

**3.论文实验为什么能提升效果？**

作者认为提升主要来自三个方面：

第一，统一管理长期记忆和短期记忆；

第二，能够根据上下文动态调度记忆；

第三，引入了生命周期机制，使重要记忆得到强化，不重要记忆被逐渐淘汰。
