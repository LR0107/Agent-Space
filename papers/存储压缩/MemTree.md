# From Isolated Conversations to Hierarchical Schemas: Dynamic Tree Memory Representation for LLMs

**NeurIPS 2024**

提出一种新的长期记忆机制 **MemTree**，用动态树结构来组织和管理大模型记忆，使模型能够更好地处理长对话、多文档推理和长期知识积累。

## 一、研究背景

### LLM 的长期记忆问题

虽然 GPT-4、Gemini 等模型的上下文窗口越来越大，甚至能达到百万级 Token，但它们仍然存在一个核心问题：**不会有效管理长期记忆。**

模型虽然“看得到”大量历史信息，但：

- 历史信息越来越长；
- 重要信息被埋没；
- 多个相关事实分散在不同位置；
- 推理时难以把这些信息整合起来。

目前大部分记忆增强方法都采用：数据库或向量库的形式存储。

这种方式的问题是：

**问题1：记忆之间没有联系**

例如：

```
2024.8：我住在旧金山
2024.9：我搬到了西雅图
```

系统会保存成两条独立记录。

但人类会自动形成：“这个人曾经住旧金山，现在住西雅图。”

**问题2：难以进行复杂推理**

例如问：“你以前住在哪里？”

系统需要：

1. 找到旧金山记录
2. 找到西雅图记录
3. 判断时间顺序
4. 推理出变化过程

普通向量检索经常做不好。

论文借鉴认知科学中的 **Schema理论**。

人脑不会把经历当作孤立事件存储。而是形成层次结构：

```
旅行
├── 日本旅行
│    ├── 东京
│    └── 京都
│
└── 欧洲旅行
     ├── 法国
     └── 德国
```

新的经历会不断更新这个结构。

作者认为：

> LLM 也应该拥有类似的层次化记忆。

---

## 二、研究方法

MemTree 用一棵动态生长的树表示记忆。

**树节点包含什么？**

每个节点保存：

文本内容

例如：我住在旧金山

Embedding

文本向量表示：e = Embedding(text) 用于计算相似度。

父节点parent

子节点children

深度depth：表示位于树的第几层。

假设系统已经有：

根节点

├── 美食
│    └── 我喜欢意大利面

└── 居住地
     └── 我住在旧金山

现在来了新信息：我搬到了西雅图

**Step1：从根节点开始**

计算：新信息vs各个子节点的语义相似度。

**Step2：选择最相似分支**

与居住地更接近。

于是进入该分支。

**Step3：继续向下搜索**

比较：我搬到了西雅图和我住在旧金山的相似度。

**Step4：判断是否属于同一主题**

如果：Similarity > Threshold

只选择相似度最高的那个分支继续向下。

否则：新建节点

成为新的叶子节点。

**Step5：更新父节点摘要**

假设叶子节点：

```
我住在旧金山
我搬到了西雅图
```

父节点不会简单复制。

而是让 LLM 生成更抽象的总结：

```
用户居住地发生变化，从旧金山迁移到西雅图
```

随着子节点越来越多：

```
美国城市迁移经历
```

再进一步：

```
个人生活经历
```

抽象程度不断提高。

### 为什么树结构有优势？

自动聚类：相似内容自然聚在一起。

多层抽象：越往上越概括、越往下越具体。

便于推理：

例如问：“最近几年生活有什么变化？”

可以直接访问高层节点。不用遍历全部历史。

### 检索（Retrieval）怎么做？

很多树结构方法需要逐层搜索。

MemTree 没这么做。

作者采用了：Collapsed Tree Retrieval——把整棵树“拍扁”

全部节点一起参与检索。

步骤：

1.问题编码

```
q → embedding
```

2.与所有节点计算相似度

```
cos(q, node)
```

3.筛选：删除低于阈值的节点。

4.Top-K：返回最相关节点。

这样既保留树结构知识，又避免搜索过程太复杂。

---

### 工作流程

**1.新信息进入**

用户的新对话先被转换成向量（Embedding）。

**2.在树中寻找位置**

从根节点开始，根据语义相似度找到最合适的分支：

- 相似 → 放到已有主题下面；
- 不相似 → 创建新的节点。

**3.更新树结构**

插入后，沿途父节点会自动生成更高层次的摘要，使树逐渐形成

上层保存概括知识，下层保存具体细节。

**4.用户提问**

问题也会被转换成向量。

**5.检索相关记忆**

计算问题与所有节点的相似度，选出最相关的几个节点（Top-K）。

**6.生成答案**

将检索到的记忆发送给 LLM，由模型生成最终回答

## 三、实验设计

作者设计了三个实验。

### 实验1：MSC（Multi-Session Chat）

任务：长对话记忆测试。

共有：

- 500组会话
- 每组约15轮对话

之后问问题：考察模型能否记住历史信息。

对比方法

- GPT-4o + 全历史
- MemGPT
- MemoryStream
- MemTree

结果：

| 方法           | Accuracy |
| ------------ | -------- |
| 全历史输入        | 95.6     |
| MemGPT       | 70.4     |
| MemoryStream | 84.4     |
| MemTree      | 84.8     |

### 实验2：MSC-E（扩展长对话）

作者把对话扩展到：200轮模拟真正长期聊天。

结果：

| 方法           | Accuracy |
| ------------ | -------- |
| 全历史输入        | 78.0     |
| MemoryStream | 80.7     |
| MemTree      | 82.5     |

### 实验3：MultiHop RAG

**数据集**

609篇新闻、2556个问题。

问题分三类：

Inference：根据文章推断结果

Comparison：A和B谁更高？

Temporal时间推理类：哪个事件先发生？

**为什么难？**

因为答案通常分散在多个文档里。必须整合信息。

结果：

| 方法           | Overall |
| ------------ | ------- |
| MemoryStream | 74.7    |
| MemTree      | 79.4    |
| RAPTOR(离线树)  | 81.0    |

比较推理：64.8 → 74.1

提升约：9.3个百分点

时间推理：59.3 → 65.3

提升约：6个百分点

## 四、总结

### 创新点

MemTree首次把记忆组织成层次结构

### 不足

非常依赖 LLM 摘要：父节点内容完全由 LLM 生成。

一个节点只能选择一个父节点：会造成知识归属错误

树的价值主要体现在组织知识，而不是加速检索

---

### Q&A

**1.为什么不用 Graph（图）而要用 Tree（树）？**

树结构实现简单、更新效率高、插入复杂度低，适合实时系统。但缺点是一个节点只能属于一个父节点。

在图结构中可以同时连接多个主题，而树结构只能选择一个分支。因此从知识表达能力看，Graph 比 Tree 更强，但工程复杂度也更高。

**2.MemTree 的核心创新是什么？**

我认为最重要的创新不是树，而是在线层次化记忆构建（Online Hierarchical Memory Building）。以前的方法只是存储和检索记忆，而 MemTree 在每次插入新信息时都会更新高层摘要，使知识结构不断演化。

**3.父节点摘要是怎么生成的？**

父节点摘要由 LLM 完成。当新信息插入后，系统会把：

- 当前父节点摘要
- 新增内容

一起输入给 LLM，让模型生成新的摘要。

**4.这种摘要会不会导致信息丢失？**

虽然叶子节点仍然保留原始内容，但父节点摘要可能遗漏细节，甚至产生幻觉。因此系统性能很大程度上依赖 LLM 的总结质量。

**5.为什么检索时不沿着树搜索？**

论文采用的是把整棵树拍平，对所有节点进行向量检索。

原因是树搜索容易走错路径。例如问题可能同时涉及多个主题分支，如果只沿一条路径搜索，容易遗漏信息。因此作者保留树用于组织知识，而检索仍然采用成熟的向量搜索方式。

**6.既然检索时拍平了，树还有意义吗？**

有。树的价值不在于加速检索，而在于构建高层语义。

普通向量库只能检索具体事实；而 MemTree 可以额外检索抽象知识。

因此树的主要作用是知识组织与知识压缩，而不是索引结构。
