# Lightweight LLM Agent Memory with Small Language Models

**ACL 2026**

## 一.研究背景

普通大语言模型虽然可以处理对话、调用工具、完成复杂任务，但在长期交互中存在一个核心问题：模型上下文窗口有限，无法稳定记住很久以前的对话内容、用户偏好、历史决策或跨轮任务信息。

因此，很多 LLM Agent 系统会引入外部记忆系统，让模型能够在需要时检索过去的信息。

论文指出，现有记忆系统主要有两类：

**1.基于检索的外部记忆系统**

MemoryBank、ReadAgent 等。

这类方法通常会把历史对话压缩成若干记忆条目，然后通过向量相似度检索 Top-K 相关记忆。

**优点：**

- 检索速度快；
- 易部署。

**缺点：**

- 只靠向量相似度容易召回噪声；
- 候选记忆过滤不稳定；
- 长期对话中回答准确率波动较大。

**2.基于大模型驱动的记忆系统**

MemGPT、A-MEM 等。

这类方法会反复调用大模型来进行记忆写入、检索、控制和更新。

**优点：**

- 语义理解能力更强；
- 记忆管理更智能；
- 准确率通常更高。

**缺点：**

- 多次调用大模型导致延迟高；
- 长对话中计算成本不断积累；
- 在线交互体验受影响。

所以，本文要解决的是：如何在保持较高记忆准确率的同时，降低在线记忆系统的计算开销和延迟？

作者提出：用小模型 来承担高频、结构化的在线记忆操作；把复杂的长期抽象和整合放到离线阶段完成。

---

## 二.研究方法

核心思想是：把记忆系统拆成多个模块，并让不同的小模型负责不同的轻量级任务，从而实现高效、低延迟、可扩展的 Agent Memory。

三层记忆结构

| 记忆类型 | 作用             |
| ---- | -------------- |
| 短期记忆 | 当前上下文窗口里的最近对话  |
| 中期记忆 | 保存用户相关的历史交互摘要  |
| 长期记忆 | 离线整合后的去身份化知识图谱 |

**1.STM：短期记忆**

STM 就是当前对话窗口里的内容。

它不持久化，也不单独检索，只作为当前会话的临时上下文。

**2.MTM：中期记忆**

MTM 是 LightMem 中最重要的个性化记忆层。

它保存的是用户相关的交互摘要，例如：

- 用户偏好；
- 历史选择；
- 任务进展；
- 曾经表达过的约束；
- 对话中的重要事实。

每条 MTM 记忆包含：

- 简短语义摘要；
- 时间信息；
- 访问统计；
- 向量 embedding；
- 用户标识符。

用户标识符用于实现多用户环境下的隔离，避免不同用户的记忆混用。

3.LTM：长期记忆

LTM 保存的是经过离线整理后的长期知识。

它不是原始用户对话，也不包含用户身份信息，而是：

- 稳定事实；
- 通用知识；
- 跨任务规律；

论文将 LTM 组织为一个轻量级的图结构知识库，这样可以支持多跳推理和知识关联。

---

### 关键模块

都是Llama-3.2-1B-Instruct（Ollama 本地部署），SLM-2经过 LoRA 微调

**1.SLM-1：检索控制器**

SLM-1 不直接回答问题，它负责把用户输入转换成检索计划。

它主要做三件事：

第一，判断用户问题的意图

例如：

- 这个问题是否依赖用户个人历史？
- 是否需要最近的对话信息？
- 是否需要长期稳定知识？
- 是否需要个性化回答？

第二，生成 Hypothetical Queries（HQs）

HQ 可以理解为“用于检索的改写查询”。

例如用户问：“推荐一个晚餐地点。”

SLM-1 可能会生成两个查询：

1. 查询用户过去喜欢什么菜系、是否有饮食限制；
2. 查询附近评分较高的餐厅。

第一个查询路由到 MTM，第二个查询路由到 LTM。

第三，分配 Top-K 检索预算

LightMem 设置一个固定检索预算 K。

SLM-1 要决定：

- 从 MTM 检索多少条；
- 从 LTM 检索多少条；
- 哪些查询需要更多预算。

这样可以避免无限扩展检索范围，保证在线延迟可控。

**2.两阶段检索机制**

**Stage 1：向量粗检索**

系统先根据 SLM-1 生成的多个 HQ，在 MTM 和 LTM 中进行向量检索。

这一阶段主要追求 召回率，即先尽量找到可能相关的记忆。

论文设置了一个“双倍预算”：

```
最终需要 K 条记忆
第一阶段先召回 2K 条候选记忆
```

也就是说，先多找一些再过滤。

**Stage 2：语义过滤与压缩**

然后由 SLM-2 对这 2K 条候选记忆进行语义判断。

SLM-2 不生成答案，只做筛选：

- 哪些记忆真正支持用户问题？
- 哪些只是表面相似但语义无关？
- 哪些记忆应该被丢弃？
- 最终保留不超过 K 条。

这个步骤的作用是降低向量检索带来的噪声。

**3.SLM-3：记忆写入**

在系统生成回答后，SLM-3 会从当前交互中提取有长期价值的信息，并写入 MTM。

它不会保存完整聊天记录，而是压缩成简短记忆摘要。

例如一段对话可能被写成：

> 用户偏好安静、价格适中的日式餐厅。

SLM-3 还负责维护 MTM：

- 合并重复记忆；
- 处理冲突信息；
- 删除过时或低价值记忆；
- 保持 MTM 容量不超过上限 。

论文中设置 MTM 容量上限为：每个用户最多保留约一万条中期记忆。

**4.离线长期整合**

LightMem 把复杂、耗时的记忆整合放到离线完成。

离线阶段使用大上下文 LLM，把 MTM 中高价值记忆抽象成 LTM 知识。

这个过程包括：

- 选择需要整合的 MTM 条目；
- 去除用户身份信息；
- 抽象成通用语义知识；
- 插入到 LTM 图结构中；
- 合并相似节点；
- 删除低置信度或过时知识。

关键点：离线整合不会阻塞在线对话，因此不会增加用户交互时的延迟。

---

### 工作流程

用户提问

SLM-1分析问题，生成检索查询，决定查 MTM 还是 LTM

向量检索 先召回一批候选记忆

SLM-2过滤候选，只保留最相关的记忆

GPT-4o-mini 根据当前上下文 + 检索记忆生成回答

SLM-3提取本轮对话中的重要信息，写入MTM

离线 LLM 后台把高价值 MTM 整理进LTM

## 三.实验设计

### 实验设置

**数据集**

**LoCoMo**是一个用于评估长期对话记忆的 benchmark。

它包含平均约 9K tokens 的长对话，问题类型包括：

- Single-hop：单跳问题；
- Multi-hop：多跳推理；
- Temporal：时间相关问题；
- Open-domain：开放域问题；
- Adversarial：对抗性问题。

这个数据集主要检验模型是否能在长历史对话中找到相关信息。

**DialSim**是一个长期多方对话模拟数据集，来自电视剧场景。

它包含：

- 1300 个多方会话；
- 时间跨度较长，更强调长期对话理解和语义一致性。

**Baseline**

| 方法              | 特点                  |
| --------------- | ------------------- |
| LoCoMo baseline | 直接使用长上下文            |
| ReadAgent       | gist memory 检索      |
| MemoryBank      | 长期用户记忆库             |
| MemGPT          | 类操作系统式内存分页          |
| A-MEM           | 强记忆基线，使用 LLM 驱动记忆网络 |
| LightMem        | 本文方法，小模型驱动在线记忆      |

论文测试了多个不同规模的模型，包括：

- GPT-4o；
- GPT-4o-mini；
- Qwen2.5-1.5B；
- Qwen2.5-3B；
- Llama-3.2-1B；
- Llama-3.2-3B。

为了公平比较，LightMem 和各基线使用相同的设置。

实现细节：

- 生成器：GPT-4o-mini；
- SLM 模块：量化版 Llama-3.2-1B-Instruct；
- embedding 模型：all-MiniLM-L6-v2；
- 第一阶段粗检索：Top-10；
- SLM-2 使用 LoRA 微调。

**评价指标**

| 指标                | 作用        |
| ----------------- | --------- |
| F1                | 衡量答案正确性   |
| BLEU-1            | 衡量词级重合    |
| ROUGE-L           | 衡量序列级重合   |
| ROUGE-2           | 衡量二元词组重合  |
| METEOR            | 考虑词形和同义匹配 |
| SBERT similarity  | 衡量语义相似度   |
| Latency P50 / P95 | 衡量系统延迟    |

其中，DialSim 特别使用 SBERT similarity。 

### 实验结果

1.在 LoCoMo 上，LightMem 在多个模型设置下都优于主要基线。

LightMem 相比 A-MEM 平均 F1 提升约 **2.5**。

2.在 DialSim 上，LightMem 也取得最好结果。

其中 SBERT 相似度从 A-MEM 的 **19.51** 提升到 **23.40**。

3.相比 A-MEM，LightMem延迟：

```
P50：83 ms vs 856 ms
P95：167 ms vs 1583 ms
```

LightMem 明显更快。

## 四.总结

### 创新点

在线 / 离线解耦的“分层记忆架构：把“高频在线操作”与“低频复杂整合”拆开

用小模型替代大模型控制记忆：每一步都调用大模型 → 慢 + 贵

### 不足

没有直接评估“任务完成率 / agent success rate”

MTM/LTM 边界依然模糊：分类依赖 SLM-1 routing + heuristic

---

### Q&A

**1.LightMem 和 MemGPT / A-MEM 最大区别是什么？**

MemGPT 和 A-MEM 主要依赖大模型在线管理记忆（写入、检索、更新都用LLM），因此效果较好但计算成本高。而 LightMem 将这些高频操作交给小模型（SLM-1/2/3），只在离线阶段使用大模型做长期知识整合，从而显著降低在线延迟。

**2.SLM-1 / SLM-2 / SLM-3 分别做什么？**

SLM-1 负责将用户问题拆解成结构化检索查询（HQs），并决定从 MTM 还是 LTM 检索以及分配 Top-K 预算。SLM-2 负责对召回的候选记忆进行语义筛选与排序，过滤无关信息。SLM-3 负责在每轮对话后提取有价值信息并写入 MTM，实现记忆更新。

**3.为什么 SLM-2 要做 LoRA 微调？**

因为 SLM-2 的任务是“语义相关性判断”，属于监督性较强的分类/排序任务，仅靠 prompt 容易不稳定。使用 LoRA 可以让模型学习“哪些记忆真正支持回答”，提高过滤质量，同时仍保持轻量级。
