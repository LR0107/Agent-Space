# A Human-Inspired Reading Agent with Gist Memory of Very Long Contexts

**ICLR 2024**

## 一.研究背景

当前大语言模型虽然语言理解能力很强，但在处理很长的文本时有两个核心问题：

- **上下文窗口有限**  
  模型一次能读入的文本长度有限，超出窗口的内容无法直接处理。
- **长文本利用能力不稳定**  
  即使文本没有超过上下文窗口，随着输入变长，模型性能也可能下降，因为长文本中存在大量干扰信息，模型不一定能准确定位关键内容。

**人类阅读长文本的启发**

作者认为，人类阅读长文的方式和 LLM 通常处理文本的方式不同。

LLM 常见做法是把文本尽可能多地塞进上下文窗口，让模型被动处理全文。  
但人类读书、读会议记录或读长文章时，通常会：

1. **记住大意**：不会逐字记住全文，而是保留故事脉络、人物关系、主要论点等；
2. **按需回查**：回答具体问题时，如果细节记不清，会回到原文相关位置查找。

论文借鉴了心理学中的 **gist memory** 思想，即人类更倾向于保留“语义大意”而不是逐字记忆。基于这个思路，作者提出 **ReadAgent**：让 LLM 像人一样先形成压缩记忆，再根据问题回查原文细节。

---

## 二.研究方法

论文提出 **ReadAgent**。它是一个不需要重新训练模型、主要依靠 prompt 实现的长文本阅读智能体。

 **语义分页 → 每页生成 gist → 形成 gist memory → 根据问题回查原文页面 → 回答问题**

**1.Episode Pagination：语义分页**

第一步是把长文本切分成多个较小的阅读片段。论文把这些片段称为 episodes/pages。

关键： ReadAgent 不是按固定长度机械切分，而是让 LLM 判断哪里是自然的阅读暂停点。

论文中的分页 prompt 会在段落之间插入编号标签，比如 `<h5>`、`<h6>`，然后让 LLM 选择一个自然断点，并说明原因。

这样做的意义是：如果分页切在自然语义边界上，每个 page 内部信息更完整，后续生成 gist 时不容易丢失逻辑关系。

**2.Memory Gisting：生成 gist memory**

第二步是LLM对每个 page 生成一个简短版本，即 **gist**。

注：每个 page 单独调用一次 LLM。

这里的 gist 可以理解为：保留核心语义、人物关系、事件发展的“压缩记忆”。

每个 gist 会带有页码信息，例如：

```
<Page 3>本页主要讲了……
```

然后所有 gist 按原文顺序拼接，形成完整的 gist memory。

**gist memory 是一次性生成的（成本可控）**

这个设计有两个作用：

原文可能有几万甚至几十万词，但 gist memory 通常短得多，可以放进模型上下文窗口；

保留全局脉络：传统检索方法通常只找几个相关片段，容易丢失全文背景。而 gist memory 保留了全文的整体顺序和脉络。

**3.Interactive Lookup：交互式回查原文**

第三步是根据具体问题回查原文页面。

当用户提出问题时，ReadAgent 会先看 gist memory，然后判断：  哪些 page 的原文最值得重新阅读？（LLM做语义判断）

接着，它会把选中的原文 page 插回到对应 gist 的位置上，再让模型回答问题。

```
Page 1 gist
Page 2 gist
Page 3 原文  ← 被回查
Page 4 gist
Page 5 原文  ← 被回查
...
问题
```

这样既保留全文整体顺序，又让关键位置变成详细原文。

这相当于让模型拥有两层信息：

| 信息层级                    | 作用           |
| ----------------------- | ------------ |
| **gist memory**         | 提供全文脉络和粗粒度定位 |
| **retrieved raw pages** | 提供精确细节和完整原文  |

设计了两种 lookup 策略。

**ReadAgent-P：并行回查**

模型一次性选择若干页进行回查。

例如系统允许它最多查 5 页，模型可能回答：

```
I want to look up Page [3, 8, 12].
```

然后系统将第 3、8、12 页原文替换回 gist memory 中对应位置，最后让模型作答。

优点：

- 调用次数少、成本较低；
- 适合问题比较明确、相关页面容易判断的任务。

缺点：

- 一次性判断可能选错、没有机会根据已查内容继续调整方向。

**ReadAgent-S：顺序回查**

模型每次只能选择一页，读完之后再决定是否继续查下一页，直到达到上限或主动停止。

这更接近人类阅读：先翻到可能相关的地方，看完发现还需要另一处信息，再继续翻。

优点：

- 更灵活；
- 适合信息分散、结构混乱的文本；

缺点：

- 需要多轮 LLM 调用；
- 成本高于 ReadAgent-P。

实验发现ReadAgent-S 明显优于 ReadAgent-P

---

### 计算成本

ReadAgent 的额外成本主要来自三块：

1. **分页成本**：需要 LLM 多次判断断点；
2. **gisting 成本**：每个 page 都要调用 LLM 生成 gist；
3. **lookup 成本**：回答每个问题时还要调用 LLM 选择页面。

但论文指出，gisting 是一次性成本；如果同一篇文档要回答多个问题，这部分成本可以被摊销。lookup 和回答阶段主要处理 gist memory，而不是全文，所以比反复输入全文更省。

---

### 工作流程

1.语义分页：把长文本切成多个页面

ReadAgent 先让 LLM 阅读长文本，并判断哪里适合暂停。

它不是机械地按固定长度切分，而是选择自然断点，例如：

- 一个情节结束；
- 一段对话结束；
- 一个论点结束；
- 主题发生转换。

切分后的每个片段被称为一个 page 或 episode。

**2.生成 Gist Memory：为每页写简短记忆**

接着，ReadAgent 会让 LLM 把每一页压缩成一个简短的 gist，也就是该页的大意。

这些 gist 会带上页码，并按原文顺序拼接起来，形成 **gist memory**。

可以理解为：gist memory = 带页码的全文阅读笔记

它比原文短很多，但保留了全文的整体脉络。

**3.按需回查：根据问题翻回原文**

当需要回答问题时，ReadAgent 会先阅读 gist memory，然后判断哪些页面可能包含答案。

之后它会回查这些页面的原文，再结合：

- 全局 gist memory；
- 被查回来的原文细节；

生成最终答案。

## 三.实验设计

### 实验设置

在三个长文本阅读理解任务上评估 ReadAgent：

| 数据集             | 任务类型      | 特点                    |
| --------------- | --------- | --------------------- |
| **QuALITY**     | 长文章多选题    | 文本较长，但多数可以放入 8K 上下文窗口 |
| **NarrativeQA** | 小说、电影剧本问答 | 文本极长，平均长度达到几万词        |
| **QMSum**       | 会议记录问答与摘要 | 会议文本长，结构较松散           |

 **模型**

 PaLM 2-L，其上下文长度为 **8K tokens**。

**Baseline**

作者将 ReadAgent 与多种 baseline 比较：

| 方法                            | 含义                    |
| ----------------------------- | --------------------- |
| **Full Raw Content**          | 直接把全文输入模型             |
| **Truncated Raw Content**     | 文本太长时，只截取部分内容         |
| **GistMem**                   | 只使用 gist memory，不回查原文 |
| **BM25 Retrieval**            | 使用关键词检索相关页面           |
| **Neural Retrieval**          | 使用向量相似度检索相关页面         |
| **ReadAgent-P / ReadAgent-S** | 本文提出的方法               |

**评价指标**

不同任务使用不同评价指标：

| 数据集             | 指标               |
| --------------- | ---------------- |
| **QuALITY**     | Accuracy         |
| **NarrativeQA** | ROUGE、LLM Rating |
| **QMSum**       | ROUGE、LLM Rating |

LLM Rating 是用另一个 LLM 判断模型答案与标准答案是否一致。

### 实验结果

- 在 QuALITY 上，ReadAgent 压缩率为 85.53%，准确率超过直接读取全文的方法；
- 在 NarrativeQA 上，ReadAgent 能处理远超模型上下文窗口的小说和剧本，最高将有效上下文长度扩展约 **20 倍**；
- 在 QMSum上，ReadAgent-S 表现最好，说明逐步回查对结构松散的会议文本更有效；

---

## 四.总结

### 创新点

提出了“gist memory + 原文回查”的长文本处理框架

不改模型结构、不训练，也能扩展有效上下文

### 不足

gist memory 仍然受上下文窗口限制

压缩会丢失信息：gist memory 的本质是压缩。  只要压缩，就可能丢失细节。

多次 LLM 调用带来额外成本。ReadAgent 需要多个阶段：

1. 分页；
2. 每页生成 gist；
3. 根据问题选择页面；
4. 读取原文页面后回答。

如果使用 ReadAgent-S，还需要多轮 sequential lookup。

---

### Q&A

**1.gist memory 和普通 summary 有什么区别？**

普通 summary 通常是对全文整体压缩，容易丢失结构和定位信息。ReadAgent 的 gist memory 是按页生成、带页码、保持原文顺序的压缩记忆。它不仅保留大意，还能告诉模型某类信息大概出现在第几页，方便后续回查原文。

**2.ReadAgent 和传统 RAG 有什么区别？**

传统 RAG 通常根据问题用 BM25 或 embedding 检索相关文本块，然后让模型回答。ReadAgent 则先构建整篇文章的 gist memory，让 LLM 基于全文压缩记忆判断该查哪些页。也就是说，ReadAgent 的检索不是单纯相似度匹配，而是利用 LLM 对全文结构的理解来做查找决策。

**3.为什么需要语义分页，而不是固定长度切分？**

语义分页能让每个 page 内部内容更完整，比如一个场景、一个话题尽量不被切断。固定长度切分可能把关键逻辑拆开，影响 gist 质量和后续 lookup。论文的消融实验也显示，LLM 语义分页比均匀切分表现更好。
