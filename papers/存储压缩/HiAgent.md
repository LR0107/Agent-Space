# HiAgent: Hierarchical Working Memory Management for Solving Long-Horizon Agent Tasks with Large Language Model

**ACL 2025**

提出了一种新的 Agent 记忆管理框架 HiAgent，核心思想是：让大语言模型像人一样，通过“子目标（Subgoal）”来组织工作记忆，而不是把所有历史信息一股脑塞进上下文。

## 一、研究背景

当前很多 LLM Agent（如 ReAct）将每一步结果全部保留在上下文里。

**问题1：上下文越来越长**

很多早期细节实际上已经不重要了。

**问题2：大量冗余信息**

模型需要在几千甚至上万 token 中寻找：注意力被稀释。

**问题3：推理能力下降**

论文引用了 Long Context 的研究：

> 当上下文很长时，LLM容易遗忘中间的重要信息。

结果：

- 行动错误增加
- 重复探索
- 任务失败

**人类是怎么解决的？**

作者借鉴了认知科学中的 Chunking（组块化）理论

例如搬家任务

人不会记住：

```
第1分钟做了什么
第2分钟做了什么
```

而会记成：

```
子任务1：把卧室收拾好
子任务2：把书打包子
```

完成后只记住：

```
卧室已整理
书已打包
```

而不是所有细节。

## 二、研究方法

用“子目标（Subgoal）”作为工作记忆的基本单元。

标准 Agent工作记忆：

```
动作1 + 观察1
动作2 + 观察2...
```

全部保留。

HiAgent工作记忆：

```
子目标1→总结结果1
子目标2→总结结果2
```

**Step1：先生成子目标**

传统Agent：直接生成动作

HiAgent先想：下一阶段目标是什么？再执行动作

**Step2：完成子目标**

执行：

```
Action
Observation..
```

直到子目标完成。

**Step3：观察总结**

完成子目标后：

原来：

```
动作1 观察1
动作2 观察2
```

全部保留。

**没有删除完整轨迹，保存在外部。**

HiAgent把它压缩成：一个总结（LLM做摘要）。

这样：

工作记忆（直接放进 LLM Prompt）从几十条轨迹

变成一句话——大幅降低上下文长度。

**Step4：轨迹检索**

如果总结丢失了重要信息怎么办？

作者设计了：Retrieval模块

当模型需要Subgoal 2

系统可以重新取出：

```
动作1观察1
动作2观察2...
```

完整历史。

---

### 工作流程

先规划子目标，再执行；完成后压缩记忆；需要时再恢复细节。

- **生成子目标**
  - 在执行动作前，先规划当前阶段要完成的子目标。
- **执行动作**
  - 围绕当前子目标持续执行 Action → Observation。
- **总结并压缩记忆**
  - 子目标完成后，将该阶段的详细轨迹总结成简短结果，只保留“子目标 + 总结”，减少上下文长度。
- **按需检索**
  - 如果后续需要历史细节，可以从记忆中检索并恢复对应子目标的完整轨迹。

| 功能      | 谁完成 |
| ------- | --- |
| 子目标生成   | LLM |
| 动作生成    | LLM |
| 子目标完成判断 | LLM |
| 轨迹摘要    | LLM |
| 是否检索历史  | LLM |
| 检索哪个子目标 | LLM |

## 四、总结

### 创新点

首次把 Subgoal 当作 Working Memory 的组织单元，更符合人类解决问题的方式。

层次化工作记忆管理，按需恢复历史轨迹

### 不足

检索机制设计得比较粗糙 ，LLM决定检索哪个Subgoal

实验规模偏小：只测了5个任务，而且都来自AgentBoard

全部依赖 GPT-4，不知道换成Llama、Qwen是否还能保持同样收益。

**Subgoal质量决定上限 ：整个框架建立在Subgoal正确基础上。**

### Q&A

**1.为什么压缩记忆不会损失重要信息？**

因为 HiAgent 采用的是“摘要 + 检索”机制。完成的子目标只在 Prompt 中保留摘要，但完整轨迹仍然存储在外部。当模型认为需要查看历史细节时，可以通过 Trajectory Retrieval 重新恢复对应子目标的完整执行过程，因此信息并没有真正丢失。

**2.子目标是谁生成的？**

子目标由 LLM 动态生成。HiAgent 没有独立的 Planner 模块，而是在每个阶段先让 GPT-4 生成当前子目标，再围绕该子目标执行动作。

**3.如果生成了错误的子目标怎么办？**

这是 HiAgent 的一个潜在局限。整个框架建立在子目标合理的前提下，如果 LLM 规划出错误或不合理的子目标，那么后续的执行、摘要和检索都会受到影响。论文并没有专门分析错误子目标的影响。
