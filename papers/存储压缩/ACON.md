# ACON: Optimizing Context Compression for Long-horizon LLM Agents

**ICML 2026**

## 一.研究背景

现在很多 LLM Agent 不是只回答一个问题，而是要在环境里连续执行很多步，比如：

- 操作邮件、表格、文档；
- 调用搜索工具；
- 使用多个 App 完成用户任务；
- 根据前面操作结果继续决策。

这类任务叫 **long-horizon agentic tasks**，也就是长周期、多步骤的智能体任务。

问题在于，Agent 每一步都会产生新的内容：

```
用户任务 → 观察环境 → 思考 → 执行动作 → 得到反馈 → 再思考 → 再执行……
```

这些内容会不断累积到上下文里，包括：

- 历史动作；
- 环境观察；
- 工具返回结果；
- 文件路径、API 参数、账户状态、任务进度等。

文章指出，这会带来两个主要问题：

**1.上下文越来越长，推理成本越来越高**

Transformer 模型处理长上下文时，需要更多计算和 KV cache 显存。长任务跑到后面，输入 token 会越来越多，成本明显上升。

**2.太多无关信息会干扰模型判断**

长上下文里并不都是有用信息。旧的工具输出、冗余观察、无关细节可能会干扰模型，使 Agent 反而更容易做错决策。

所以，文章要解决的问题是：**如何压缩 Agent 的上下文，同时保留真正影响后续决策的关键信息？**

### 已有方法的问题

**1. 简单截断不可靠**

比如只保留最近几轮历史，虽然省 token，但可能会删掉前面很关键的信息。

**2.普通摘要太泛化**

直接让 LLM “总结一下历史”也不够可靠。因为 Agent 场景里重要信息非常复杂，包括：

- 当前任务进度；
- 工具调用结果；
- 中间变量；
- 环境状态变化；
- 失败原因；
- 后续必须使用的参数。

普通摘要容易漏掉这些结构化状态信息。

**3.一些方法需要微调模型，不适合 API 模型**

有些方法通过模型参数更新来训练压缩器，但这对 GPT-4.1 这类闭源 API 模型不现实，因为用户不能修改模型权重。

因此，文章提出：**不要改模型参数，而是优化压缩提示词。**

## 二.研究方法

一个面向长任务 Agent 的上下文压缩框架，通过自动优化“压缩指南”，让压缩器知道哪些信息必须保留、哪些信息可以删掉。

ACON 不是简单让模型“总结一下”，而是先学习一套更好的压缩规则。

关键是：如果 Agent 在完整上下文下能成功，但在压缩上下文下失败，说明压缩过程中可能丢掉了关键内容。

于是 ACON 会比较两条轨迹：

```
完整上下文轨迹：成功
压缩上下文轨迹：失败
```

然后让一个 LLM 分析：

- 压缩后丢了什么重要信息？
- 哪些变量、状态、动作结果没有保留？
- 哪些摘要写得不够准确？
- 哪些内容导致 Agent 后续决策错误？

这些分析结果会被用来更新压缩指南。

### 工作流程

**第一步：先跑 Agent，收集轨迹**

在训练任务集上，分别运行：

1. 不压缩上下文的 Agent；
2. 使用当前压缩规则的 Agent。

然后找出一种特殊样本：

```
不压缩时成功，压缩后失败
```

这些样本最有价值，因为它们能暴露压缩器的问题。

**第二步：失败分析，生成反馈**

ACON 让 **LLM** 对比完整上下文和压缩上下文，分析失败原因。

例如反馈可能是：

```
压缩摘要中遗漏了用户指定的文件路径；
历史中删除了前一步失败原因，导致 Agent 重复错误操作。
```

文章把这种反馈称为一种 natural language gradient，也就是“自然语言形式的梯度”。

意思是：虽然不能像神经网络那样反向传播，但可以用自然语言告诉压缩提示词应该怎么改。

**第三步：优化压缩指南**

根据多个失败样本的反馈，**LLM** 会生成新的压缩指南。

例如原来的压缩提示词可能只是：

```
请总结历史内容，保留重要信息。
```

优化后可能变成更结构化的规则：

```
你需要维护一个状态保持型历史摘要，包括：
1. 当前任务目标；2. 已完成步骤；3. 关键变量和值；4. 工具调用结果；5. 后续必须继续使用的参数；6. 失败尝试及其原因。
```

这样压缩器就不只是“写摘要”，而是按照 Agent 任务需求保留状态。

**第四步：蒸馏到小模型，降低压缩器成本**

如果每次压缩都调用大模型，成本还是很高。

所以文章进一步提出：把优化后的压缩器蒸馏到小模型上。

具体来说：

1. 用大模型 + 优化后的压缩指南生成压缩结果；
2. 把这些输入输出对作为训练数据；
3. 用 LoRA 等方法微调小模型；
4. 推理时用小模型负责压缩，大模型只负责 Agent 决策。

这样可以减少压缩模块本身的计算开销。

## 三.实验设计

### 实验设置

**Benchmark**

| Benchmark          | 任务特点                                     | 评价指标                   |
| ------------------ | ---------------------------------------- | ---------------------- |
| **AppWorld**       | 模拟多个日常 App，例如 Spotify、Venmo、SimpleNote 等 | 任务完成得分 / Accuracy      |
| **OfficeBench**    | 办公场景任务，涉及 Word、Excel、Email 等             | benchmark 定义的 accuracy |
| **8-objective QA** | Agent 使用搜索工具回答 8 个问题                     | EM 和 F1                |

**Baseline**

| 方法                 | 含义                         |
| ------------------ | -------------------------- |
| **No Compression** | 不压缩，保留完整上下文                |
| **FIFO**           | 只保留最近 k 轮历史                |
| **Retrieval**      | 根据相似度检索相关历史                |
| **LLMLingua**      | 使用已有 prompt compression 方法 |
| **Prompting**      | 普通 LLM 摘要压缩                |

### 实验结果

**1.ACON 能明显减少 token**

ACON 在多个 benchmark 上可以减少 **26%–54% 的 peak token usage**。

**2.ACON 不只是省 token，还能提升成功率**

原因是：压缩不仅减少了成本，也去掉了很多干扰信息，让模型更专注于关键状态。

尤其是小模型更容易被长上下文干扰，所以压缩后提升更明显。

**3.小模型压缩器效果接近大模型**

文章把 GPT-4.1 压缩器蒸馏到 Qwen3-14B、Qwen3-8B、Phi-4 等小模型上。

结果显示，小模型压缩器可以保留大模型教师压缩器 **95% 以上的性能**。

这说明压缩任务本身不一定需要大模型长期承担，小模型可以胜任。

**4.ACON 对小模型 Agent 帮助很大**

文章还测试了小模型 Agent。结果显示，ACON 可以显著提升小模型在长任务中的表现：

- AppWorld 上相对提升约 **32%**；
- OfficeBench 上提升约 **20%**；
- Multi-objective QA 上提升约 **46%**。

这说明 ACON 不只是降低成本，还能让较小模型更适合做长任务 Agent。

## 四.总结

### 创新点

不是直接做压缩，而是优化**Compression Guideline**：不需要改模型参数，可以像“编程规范”一样迭代优化

用“失败-成功对比”作为监督信号，替代 RL reward

### 不足

cost 很高（要跑完整 agent），scaling 到大 benchmark 很贵

仍依赖强 LLM

compression correctness 没有严格 guarantee。ACON 的压缩是：“LLM认为重要的信息”

论文也承认：compression 引入额外 inference step，latency 上升

compression policy 仍然 heuristic，不是严格 learned policy

---

### Q&A

**1.为什么不用 RL 或 fine-tuning 来优化压缩器？**

理论上可以用 RL，但论文指出三个实际问题：第一，reward 是稀疏的，只有最终任务成功/失败，无法提供细粒度信号；第二，成本极高，因为每次训练都要跑多步 agent 交互；第三，在 API-based LLM（如 GPT-4.1）上无法更新参数。因此 ACON 选择在“自然语言空间优化 prompt”，用 LLM 自身生成反馈，相当于用文本梯度替代 policy gradient，在工程上更可行。

**2.distillation（蒸馏小模型压缩器）的意义是什么？**

如果每一步 compression 都调用 GPT-4.1，会显著增加 inference cost。distillation 的作用是把“优化后的压缩行为”固化到小模型中，使 compression 从 LLM API 调用变成轻量 forward pass。最终系统结构变成：大模型负责 decision-making，小模型负责 context compression。

**3.ACON 和 memory system / RAG / long-context model 有什么关系？**
ACON 本质上属于“context management layer”，而不是 memory architecture 或 model architecture。和 RAG 不同，它不是“外部检索补充信息”，而是“对已有 context 做结构化重写”。和 long-context model 不同，它不依赖扩展 context window，而是通过压缩控制输入规模。因此它可以与 RAG、long-context model 叠加使用。
