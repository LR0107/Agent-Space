# Benchmarking Agent Memory in Interdependent Multi-Session Agentic Tasks

ICML 2026 

## 一.研究背景

**1.只测“记住了没有”，不测“能不能用”**

很多记忆 benchmark，比如 LoCoMo、LongMemEval 等，主要测试模型能否从长对话或长文本中回忆事实。

例如：

> “之前用户说自己喜欢什么？”  
> “某段对话里提到的名字是什么？”

这种评测能看出模型的**信息检索能力**，但不能说明它在真实任务中能不能利用记忆做决策。

**2.只测单次任务执行，不需要长期记忆**

另一类 benchmark，比如 WebArena、WebShop、SWE-Bench，主要测试 Agent 的操作能力、规划能力和工具调用能力。

但这些任务通常是**单会话任务**，Agent 只需要在当前上下文中完成任务，不一定需要跨多轮、跨会话保留信息。

作者认为，这和现实不太一样。

现实中的 Agent 往往要处理这种任务：

> 第一次帮用户买了相机机身；  
> 第二次帮用户买镜头，需要记住之前买的是哪种相机；  
> 第三次帮用户买相机包，还要继续考虑兼容性。

所以，真正重要的不是“记住事实”，而是：

> **Agent 能否在多个会话中积累经验，并在后续任务中正确使用这些记忆。**

---

## 二.研究方法

核心思想是：把记忆放到完整的 **Memory-Agent-Environment Loop** 中评测。

Agent 执行动作 → 环境给反馈 → Agent 更新记忆 → 后续任务检索记忆 → 再做决策

论文强调，记忆不应该只被看作“存储文本的地方”，而应该是 Agent 做长期任务时维持状态、继承经验、指导决策的关键组件。

### 多会话 + 子任务依赖

MEMORYARENA 的任务不是独立的，而是由多个**互相依赖的子任务**组成。

一个完整任务通常包含多个 session，每个 session 对应一个子任务。后面的子任务依赖前面子任务产生的信息。

例如：

```
Session 1：购买相机机身
Session 2：购买兼容的镜头
Session 3：购买兼容的相机包
Session 4：购买配套配件
```

后面的任务不会把前面信息重新完整告诉 Agent，Agent 必须依赖自己的记忆系统。

这就逼迫 Agent 解决三个问题：

1. **该记什么？**  
   例如买了哪款相机、价格、品牌、型号。
2. **什么时候检索？**  
   后续买镜头时，要主动检索之前买的相机型号。
3. **怎么用记忆做决策？**  
   不是简单复述，而是判断兼容性、约束关系和最终选择。

### 任务环境

论文构建了四类环境，共覆盖 766 个任务。

**1.Bundled Web Shopping：组合购物任务**

模拟用户分多次购买一组相关商品。

例如：

```
先买相机机身；再买镜头；再买相机包；再买其他配件。
```

后续商品必须和前面买过的商品兼容。

这个任务主要测试：

- 商品属性记忆；
- 兼容性判断；
- 多步选择约束；
- 避免被干扰选项误导。

**2.Group Travel Planning：多人旅行规划**

模拟多人陆续加入旅行计划。

第一个人先确定基础行程，后面的人加入时会提出新偏好。

例如：

```
Tracy 先规划了 5 天游；
Chelsea 后来加入，她想住比 Tracy 第一天酒店贵 150 美元的酒店；
Emily 再加入，她想和 Chelsea 第三天吃同一家餐厅。
```

这类任务要求 Agent 记住：

- 每个人的偏好；
- 每天的行程安排；
- 谁和谁共享活动；
- 后续人的约束如何依赖前面人的计划。

这是论文里最难的环境之一，因为每个任务包含大量活动 slot 和交叉约束。

**3.Progressive Web Search：渐进式网页搜索**

这个任务模拟逐步增加条件的信息搜索。

例如：

```
第一步：找 1989 年获得博士学位的人；
第二步：在这些人中，找 2014 年出版过书的人；
第三步：再找其中 2012 年做过 keynote speaker 的人。
```

每一步都依赖前一步结果。

Agent 不能只看当前问题，而必须保留之前筛选出的候选集合。

这个任务主要测试：

- 跨步骤信息积累；
- 条件逐步收窄；
- 搜索结果记忆；
- 多条件组合推理。

**4.Sequential Formal Reasoning：连续形式推理**

包括数学和物理两个方向。

任务来自真实学术论文中的长推导链。

例如：

```
先证明一个中间引理；再基于这个引理推出另一个公式；最后解决一个核心理论问题。
```

这类任务不是普通数学题，而是更接近研究论文中的长链条推理。

它主要测试：

- 公式和定义记忆；
- 中间结论复用；
- 长链推导能力；
- 是否能把前面推导作为后续推理基础。

注：为子任务和最终结果提供了人工验证的答案

## 三.实验设计

作者比较了三类 Agent 记忆方式。

**1.Long-context Agent：长上下文记忆**

这种方式最直接：

> 把过去所有交互历史直接拼到 prompt 里。

代表模型包括：

- GPT-5.1-mini；
- GPT-4.1-mini；
- Gemini-3-Flash；
- Claude-Sonnet-4.5。

这种方法不做专门记忆压缩，也不建立结构化记忆，只是依赖模型长上下文能力。

**2.External Memory Agent：外部记忆系统**

这类方法会把过去经验写入外部记忆，再在后续任务中检索。

论文测试了：

- Letta / MemGPT；
- Mem0；
- Mem0-g；
- ReasoningBank。

这类方法通常会进行记忆压缩、摘要、组织和更新。

**3.RAG-based Agent：检索增强记忆**

这类方法把过去内容放进检索系统，需要时检索相关片段。

论文测试了：

- BM25；
- embedding-based RAG；
- MemoRAG；
- GraphRAG。

其中 GraphRAG 属于结构化检索方法，会利用图结构组织信息。

### 评价指标

论文主要使用两个指标。

**1.SR：Task Success Rate**

表示完整任务是否全部成功。

例如组合购物任务中，只有最终买到的整套商品都兼容，才算成功。

这个指标很严格。

**2. PS：Progress Score**

表示任务中的子任务完成了多少。

例如一个任务有 6 个子任务，Agent 完成了 3 个，那么进度分就是 50%。

这个指标用来观察 Agent 是否至少完成了一部分任务。

### 实验结果

**1.当前 Agent 记忆能力仍然很弱**

实验结果显示，所有方法在 MEMORYARENA 上表现都不高。

尤其是：

- Group Travel Planning 几乎所有方法 SR 接近 0；
- Bundled Web Shopping 的完整成功率也很低；
- Progressive Web Search 和 Formal Reasoning 稍好，但仍存在明显失败。

这说明现有 Agent 即使在传统记忆 benchmark 上表现很好，也不代表能在真实多会话任务中有效使用记忆。

**2.外部记忆和 RAG 不一定比长上下文更好**

一个重要发现是：

> 给 Agent 加外部记忆系统，并不一定稳定提升性能。

原因主要有两个：

第一，记忆表示不匹配。 
长上下文保留的是原始交互历史，而外部记忆通常会压缩、摘要、重排信息，可能丢失关键细节。

第二，Agent 不一定会正确使用记忆。  
即使检索到了相关记忆，Agent 也可能不知道如何把它整合进当前决策。

所以外部记忆不是简单“加上就变强”。

**3.在超长任务中，外部记忆有时有帮助**

在 Progressive Web Search 和 Formal Reasoning 中，外部记忆和 RAG 有一定优势。

原因是这些任务的轨迹很长，有些超过 100k tokens。直接把全部历史塞进上下文，会导致注意力分散、错误累积。

外部记忆可以帮助压缩和提取关键信息，减轻长上下文负担。

**4.子任务越往后，Agent 越容易失败**

论文还分析了不同子任务深度下的成功率。

结果发现：

> 随着子任务深度增加，所有方法成功率都会下降。

这说明 Agent 很难持续维护跨 session 的任务状态。

---

## 四.总结

### 创新点

把“记忆评测”从事实回忆推进到任务执行：不是孤立评测记忆，而是把 Agent 放进完整交互循环中。

构造了“多会话 + 强依赖”的任务：任务不是独立的，而是多个子任务串起来，并且后面的子任务依赖前面的结果。

### 不足

任务整体难度很高，很多方法成功率接近 0，导致方法之间区分度不够。

数据构造成本高，很多任务需要人工设计和验证，尤其是数学、物理推理部分需要专家标注。

对失败原因拆解还不够细，例如没有充分区分是写入记忆失败、检索失败、理解失败，还是执行动作失败。
