# Agent如何避免翻车

**From：程序员Carl**

## 为什么会死循环

**第一，缺停止条件。**

Agent 不知道什么时候应该停。

它以为继续调用工具就能解决问题。

**第二，缺失败记忆。**

同一个工具已经失败过，它没有记住。

于是反复重试。

**第三，缺任务完成判断。**

它不知道当前信息已经不足以继续推进，应该转向追问用户。

### 怎么兜底

**第一，限制最大步数。**

比如一个客服 Agent 最多调用 5 次工具。

一个故障排查 Agent 最多跑 15 步。

超过步数，就必须总结当前进展和缺失信息。

**第二，记录失败工具。**

如果同一个工具用同一组关键参数失败过，不要无限重试。

应该改变策略：

- 缺参数，就追问用户

- 工具超时，就有限重试

- 无数据，就换查询范围

- 权限不足，就停止说明

**第三，设计停止条件。**

比如：

- 缺少关键输入

- 工具连续失败

- 达到最大步数

这几个条件一触发，Agent 就不能继续自由探索。

它要么交付阶段性结果。要么请求用户补充。要么进入人工确认。

## 上下文污染

上下文污染，是 Agent 里非常隐蔽的问题。

它不像工具调用失败那么明显。但杀伤力很大。

**上下文污染：就是错误信息、过期信息、无关信息、未验证假设，被放进上下文后，影响后续判断。**

### **来自哪里**

**第一，用户猜测。**

用户说“我觉得可能是 xx”，这不是证据。

**第二，工具返回的噪音。**

日志、搜索结果、RAG里经常有无关信息。

**第三，过期上下文。**

前面得到的结论，在后面可能已经被推翻。

**第四，模型自己的中间假设。**

模型说“可能是数据库问题”，如果不标记为假设，很容易被后面当事实。

### **怎么兜底**

上下文管理要分层。

不要把所有东西都丢进同一个上下文。

至少分成三类：

**事实。**

来自工具、数据库、日志、监控，而且能追溯来源。

**假设。**

模型推测、用户猜测、待验证的原因。

**废弃信息。**

已经被验证无关、过期或错误的信息。

Agent 最后输出结论时，必须基于事实。

假设只能作为待验证方向。

废弃信息不能继续参与推理。

这就是为什么很多成熟 Agent 系统会做状态记录、证据引用、trace 回放。

## 权限越界

前面几类翻车，大多影响答案质量。

权限越界，影响的是安全。

### 为什么会权限越界

权限越界通常来自三件事。

**第一，工具权限太大。**

一个工具既能查，又能改。

**第二，缺少确认环节。**

Agent 从“建议”直接跳到“执行”。

**第三，Host 没有把关。**

系统默认相信模型判断。

### 怎么兜底

高风险动作要分级。

可以简单分三类。

**低风险：只读查询。**

比如查文档、查订单状态、查监控指标。

通常可以让 Agent 直接调用。

**中风险：生成草稿。**

比如写邮件草稿、生成配置修改方案。

可以让 Agent 做，但不能直接提交。

**高风险：改变外部世界。**

比如发邮件、退款、删除数据、改生产配置、执行命令。

必须用户确认。

---

不要期待一个万能开关。真正的兜底是一套护栏。

至少包括六层。

**第一，任务边界。**

明确 Agent 能做什么，不能做什么。

**第二，工具边界。**

工具描述、参数、返回值、错误码都要清楚。

**第三，执行边界。**

最大步数、超时、重试次数、停止条件。

**第四，状态管理。**

记录目标、步骤、失败。

**第五，权限控制。**

高风险动作二次确认、权限校验、审计记录。

**第六，交付检查。**

最终回答前检查：目标是否完成？证据是否足够？有没有未验证假设？
